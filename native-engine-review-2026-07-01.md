## Cross-cutting observations
- The `OPTIONAL { … } FILTER(?optVar …)` idiom appears in nearly every regressed/slow query — the LeftJoin+Extension(BIND)+filter-above-optional shape defeats both the native compiler and the sketch costing (plannedFilterPassRatioConfidence=0 in plans).
- Sketch costing is the main source of both wins (geomean 1.08→2.48) and the two catastrophic outliers (ENG q8, LIBRARY q8) — cost 131 vs actual 6s indicates estimate-vs-reality blowup on aggregate-topped LeftJoin trees, not raw executor speed.
- Run noise: same-day runs differ up to 2–3x on TRAIN q8/q9, MEDICAL q1/q5, ELECTRICAL q2; treat single-run deltas <2x as noise.

## 2. Verified findings (critical/high)

### [CRITICAL][per-tuple-overhead] Full cursor-construction stack rebuilt per join probe in INLJ inner loops
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeAggregateCompiler.java:2144

**Problem.** Every left row in a nested-loop join re-opens the right side from scratch: JoinCursor.next() calls right.open(row) (LmdbNativeAggregateCompiler.java:2144; same for LeftJoinCursor:2219, ExistsFilter:2638, MinusCursor:2468). PatternPlan.open (:1678-1703) allocates a PatternCursor + PatternRowCursor, then LmdbSailDataset.statements (LmdbSailStore.java:1741-1757) acquires nativeSourceLock.readLock(), allocates a NativeSourceReadLockedRecordIterator wrapper, and TripleStore.getTriples (TripleStore.java:495-507) re-runs index selection over all indexes and constructs a new LmdbRecordIterator whose ctor (LmdbRecordIterator.java:91-153) allocates a long[4], does Pool.get() (ThreadLocal lookup), pulls 2-3 MDBVals + a key buffer from the pool, takes txnLockManager.readLock(), and calls mdb_cursor_open via JNI; all of it is torn down on exhaustion (mdb_cursor_close + pool returns + 2 lock releases). That is ~5 heap allocations, 2 lock acquire/release pairs, an index-selection loop, and 2 JNI calls of pure setup/teardown per probe. The identical pattern exists in the BGP executor: Frame.open/close per backtrack re-entry (LmdbNativeQueryCompiler.java:1078-1080, 1095 → NativePattern.open :591-615). Benchmark telemetry pins LIBRARY q8's 10-minute timeout inside this exact JoinCursor.next → PatternPlan.open chain (results digest cites LmdbNativeAggregateCompiler.java:2137/2141), and the composite-source comment itself cites 'tens of thousands of small correlated LMDB probes'.

**Recommendation.** Make the probe path reusable: (1) resolve the TripleIndex at compile time per pattern (the winning index depends only on which of s/p/o/c are bound, which is known per pattern once the join order is fixed — or at worst per bound-mask); (2) give Frame/JoinCursor a long-lived LmdbRecordIterator that keeps its mdb cursor, MDBVals and key buffer across probes and repositions with MDB_SET_RANGE on the new min key instead of close+reopen (mdb_cursor_renew/reseek on the same dbi+txn); (3) collapse the NativeSourceReadLockedRecordIterator + PatternCursor + PatternRowCursor wrappers into one reusable object whose fields are reset per probe. This removes all steady-state allocation and both lock round-trips from the innermost join loop.

**Verifier refinement.** Confirmed with corrections: (a) the fully-bound/existence-check probe path is ALREADY optimized — TripleStore keeps a ThreadLocal SpoExistenceCursor reused via mdb_cursor_renew (TripleStore.java:135-138, 684-712), so the recommendation applies specifically to binding-producing probes going through TripleStore.getTriples → new LmdbRecordIterator; (b) fully-bound LmdbRecordIterators (rangePrefixLength==4) already skip mdb_cursor_open and use mdb_get (LmdbRecordIterator.java:139-141, 248), so the 2-JNI-call setup cost applies to probes with ≥1 unbound position (the common INLJ case); (c) beyond setup/teardown, LmdbRecordIterator.next() also acquires/releases txnLockManager.readLock() per tuple (LmdbRecordIterator.java:257) — a reusable probe cursor should also amortize this; (d) compile-time index resolution must be keyed per runtime bound-mask, not per pattern, because OPTIONAL upstream can leave slots UNKNOWN at runtime (Term.lookup → UNKNOWN in PatternPlan.open). The existing SpoExistenceCursor cache is a proven in-repo template for the proposed reusable-cursor design.

### [CRITICAL][join-strategy] Aggregate compiler never pushes filters into the join: FilterPlan runs above the full join and blocks reordering
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeAggregateCompiler.java:2293

**Problem.** In the aggregate compiler, a Filter compiles to FilterPlan wrapping its arg (compileTuple :337-358 → SlotPlan.filter :1546); FilterCursor (:2325-2331) tests rows only after the entire wrapped subplan (typically the whole flattened MultiJoinPlan) has produced a joined row. There is no equivalent of the BGP compiler's assignFilters, which places each filter at the earliest depth where its requiredMask is covered (LmdbNativeQueryCompiler.java:1192-1218) and prunes per-level. Worse, FilterPlan is not reorderable (canReorder :1513-1516 accepts only PatternPlan/MultiValuePatternPlan/ValuesPlan/MultiJoinPlan), so Join(Filter(Join(a,b)), c) freezes into JoinPlan(FilterPlan(...), c) (:1500-1510) — the filter boundary also prevents global join flattening/reordering across it. Only constant equality/IN filters escape via pushdown into a statement pattern (compileFilterIntoStatementPattern :392-409). LeftJoinPlan and ExtensionPlan create the same reorder barriers.

**Recommendation.** Port the BGP path's filter machinery: give NativeBooleanFilter a requiredMask, collect filters while flattening MultiJoinPlan children (treating FilterPlan over a reorderable child as transparent: hoist the filter, keep the child reorderable), and evaluate each filter inside JoinCursor at the earliest position where its slots are bound. Same for the semi-join rewrites: a null-rejecting filter over LeftJoin already gets special-cased (:834-851), so the infrastructure for filter analysis exists.

**Verifier refinement.** The aggregate compiler lacks any per-depth filter placement (NativeBooleanFilter at LmdbNativeAggregateCompiler.java:2566 has no requiredMask; no assignFilters equivalent), and FilterPlan/LeftJoinPlan/ExtensionPlan are reorder barriers in canReorder (:1514-1517), freezing join order at the static algebra order and fragmenting MultiJoinPlan's adaptive chooseOrder. Two corrections to the description: (1) filters do not universally "run above the full join" — the upstream LmdbSketchJoinOptimizer/LmdbDeferredFilterPlacer (buildSegmentRoot, LmdbDeferredFilterPlacer.java:77-128) already interleaves filters at intermediate join positions in the algebra, and JoinPlan(FilterPlan(a⋈b), c) evaluates the filter before joining c; the general defect is loss of runtime reordering across filter boundaries plus total reliance on static placement, not full-cartesian-then-filter. (2) The escape list also includes compileJoinWithValuesFilter (:560-608, VALUES-as-filter folded into MultiValuePatternPlan) and direct EXISTS filters (:795). However, for LIBRARY q8 the claimed worst case is exact: its FILTER(?optName IN ...) over an OPTIONAL var takes the compileFactorizedLeftJoinFilter path (:834-851), yielding FilterPlan(ExistsFilter) wrapping the entire flattened 5-pattern MultiJoinPlan, so the ?member-dependent probe fires only after full row assembly although ?member binds mid-join. Recommendation stands: give NativeBooleanFilter a requiredMask, hoist FilterPlan over reorderable children during SlotPlan.join flattening (including ExistsFilter wrappers from the factorized-LeftJoin rewrite), and evaluate each filter inside MultiJoinPlan at the earliest depth where its slots are bound, mirroring LmdbNativeQueryCompiler.assignFilters (:1192-1218). Benchmark attributions (q8 >10min, PHARMA q2 740ms) could not be verified in this static review.

### [CRITICAL][correctness] FILTER(bound(?x)) / FILTER(!bound(?x)) on pattern-bound vars evaluated once at depth 0 in native BGP plans — wrong results
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeQueryCompiler.java:822

**Problem.** SlotBoundFilter.requiredMask() returns 0 (lines 821-824) and ExternalBoundFilter likewise (839-842). assignFilters places any filter whose requiredMask is covered by the initial boundMask at depth 0 (lines 1197-1200), and findNext runs acceptFilters(0) exactly once before any pattern cursor is opened (line 1065). For `SELECT * { ?x ?p ?o FILTER(bound(?x)) }`, slot ?x is UNKNOWN at that point and base has no binding, so accept() (lines 826-829) returns false and the whole iteration closes: the query returns ZERO rows where the generic evaluator returns every row (Bound is evaluated per solution via QueryValueEvaluationStep). Dually, `FILTER(!bound(?x))` compiles to NativeFilter.not(SlotBoundFilter) (line 767-769), which evaluates !false=true at depth 0 and returns ALL rows where the correct answer is zero. The filter must be anchored to the depth at which the slot is produced, not to requiredMask==0.

**Recommendation.** Make SlotBoundFilter.requiredMask() return `1L << slot` OR special-case Bound during assignFilters: place it at the depth where the slot's producing pattern has run (the slot is guaranteed bound within a BGP, so within a pure BGP fragment `bound(?x)` for a pattern var can be constant-folded to true and `!bound(?x)` to false). Keep ExternalBoundFilter at depth 0 since it only consults the base binding set.

**Verifier refinement.** Fix SlotBoundFilter.requiredMask() (LmdbNativeQueryCompiler.java:822-824) to return the existing `mask` field (`1L << slot`, already computed at line 818 but unused — evidently an oversight; compare TypeFilter at 892-895). MAX_NATIVE_SLOTS=60 keeps the shift safe. This anchors the filter (and any Not/And/Or composite built from it, since composites OR their args' masks) at the depth where the slot's producing pattern has bound it, at which point accept() is correctly constant-true for bound(?x) and NativeFilter.not yields constant-false for !bound(?x). It also fixes an unmentioned second corruption: disjunctions like FILTER(bound(?x) || sameTerm(?y,<c>)) are currently placed at ?y's depth and evaluate bound(?x) as false prematurely. ExternalBoundFilter (839-842) is correct as-is: a slotless var can only be bound by the base binding set, so mask 0 / depth 0 is right. Note the bug is masked when ?x arrives via the base binding set (initialize() copies base bindings into slots before acceptFilters(0)), so a regression test must run the query without external bindings; no existing lmdb test covers bound()/!bound() on the native path.

### [CRITICAL][correctness] Aggregate-compiler filters collapse type errors to false before Not(), so NOT IN / !(=) / !(<) over unbound OPTIONAL vars keep rows the generic engine drops
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeAggregateCompiler.java:993

**Problem.** compileBoolean(Not) is `row -> !arg.accept(row)` (lines 993-995). The leaf filters return plain false when the operand is unbound or errors: compileCompare returns false when either id is UNKNOWN (lines 1080-1090 for EQ/NE, 1094-1098 for LT/GT/...), ValueSetFilter returns false for UNKNOWN and swallows RuntimeException as false (lines 2667-2671, 2691-2693), CachedCompareFilter returns false for UNKNOWN (lines 2729-2733). SPARQL three-valued logic requires error to propagate through `!` and still yield 'row dropped': the generic engine only collapses the error at the outermost expression (QueryValueEvaluationStep.asPredicate, core/queryalgebra/evaluation/.../QueryValueEvaluationStep.java:31-40 catches ValueExprEvaluationException). Concrete failure: `?s :p ?o . OPTIONAL { ?s :q ?v } FILTER(!(?v = "x")) ... GROUP BY/COUNT` — for rows where ?v is unbound, generic evaluation raises an error inside Not and drops the row; native computes !(false)=true and keeps it (as long as "x" resolves to a store id so CachedCompareFilter is used, line 1101-1109). Same for `FILTER(?v NOT IN (<a>,<b>))` = Not(ListMemberOperator) → ValueSetFilter. The persistent LIBRARY q9 wrong-result benchmark failure (FILTER over an OPTIONAL var, 2 rows instead of 0) is consistent with this family.

**Recommendation.** Introduce three-valued results in NativeBooleanFilter (TRUE/FALSE/ERROR, e.g. an int or enum), map UNKNOWN-operand and exception cases to ERROR, implement SPARQL logic tables in And/Or/Not, and convert ERROR→reject only at the point the filter gates a row. Alternatively, refuse the native id fast-path (fall back to the generic precompiled predicate) whenever a Compare/IN filter can see a potentially-unbound slot, i.e. a slot not covered by the mandatory producedMask beneath the filter.

**Verifier refinement.** Confirmed bug, corrected scope: in LmdbNativeAggregateCompiler.compileBoolean (core/sail/lmdb/.../LmdbNativeAggregateCompiler.java:982-1025), Not/And/Or are decomposed natively while every leaf collapses unbound/type-error to plain false — Compare lambdas (1083-1084, 1094-1098), CachedCompareFilter (2730-2733), ValueSetFilter (2668-2671, 2691-2693), AND the generic leaf fallback via asPredicate (1018-1024, QueryValueEvaluationStep.java:31-40). Any FILTER containing `!(...)` over a variable that can be unbound (OPTIONAL right-side var, LeftJoinCursor null-extension at 2198-2221) therefore yields !(false)=true and keeps rows the generic engine drops (DefaultEvaluationStrategy.prepare(Not) 1344-1347 propagates ValueExprEvaluationException; collapse happens only at the outermost predicate). Triggers: `FILTER(!(?v = const))`, `FILTER(!(?v < const))`, `FILTER(!(?v IN (...)))`, `FILTER(!(?a = ?b || ...))` under GROUP BY/aggregates (default-on path, LmdbNativeEvaluationStrategy.java:42-52). NOT the trigger: `?v NOT IN (...)` and `?v != const` — the parser lowers NOT IN to Compare(NE)/And-of-NE (TupleExprBuilder.java:2546-2581), and NE over UNKNOWN drops the row in both engines, so behavior matches. Consequently the LIBRARY q9 failure (`FILTER(?optTitle != "")`, plan snapshot lmdb-library-q9-*.json) is NOT explained by this bug — q9 uses plain NE and usually the drop-consistent factorized EXISTS path (834-851); its wrong result has another cause. Fix: implement three-valued (TRUE/FALSE/ERROR) evaluation in NativeBooleanFilter with SPARQL logic tables for And/Or/Not, mapping UNKNOWN-operand, RuntimeException, and leaf-asPredicate-error cases to ERROR, converting ERROR→reject only where the filter gates a row (FilterCursor 2325-2331, ExistsFilter, LeftJoin condition). If instead using the bail-out approach, note: (a) it must reject any Not whose subtree references a potentially-unbound slot — not just Compare/IN leaves, because the generic leaf fallback also collapses errors below Not; (b) SlotPlan.producedMask cannot be used as the guard since LeftJoinPlan/UnionPlan OR in optional branches (2170, 2245) — a separate guaranteed/mandatory-bindings mask is required.

### [CRITICAL][coverage-gaps] Non-aggregate queries fall off the native path entirely even though the executor already supports OPTIONAL/UNION/MINUS/BIND/VALUES — SlotPlan is only reachable under a Group root
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeQueryCompiler.java:112

**Problem.** The BGP compiler's Builder.collect accepts only Filter/Join/StatementPattern/SingletonSet and returns false for everything else (LmdbNativeQueryCompiler.java:112-136, default bail :135), so any non-aggregate fragment containing LeftJoin, Union, Difference, Extension, BindingSetAssignment, Exists-subplans etc. cannot compile. Meanwhile the aggregate compiler's compileTuple already implements native cursors for exactly these nodes — LeftJoin (LmdbNativeAggregateCompiler.java:294-314), Union (:315-320), Difference/MINUS (:321-336), Filter (:337-359), Extension var-copy (:360-385), VALUES (:386-388, compileValues :957-980), plus native EXISTS (:997-1004) — but its tryCompile only fires when the root is Group or Filter(Group) (LmdbNativeAggregateCompiler.java:94-104). Because DefaultEvaluationStrategy.precompile recurses virtually per node (JoinQueryEvaluationStep.java:46-47; DefaultEvaluationStrategy.java:585-773), the cliff is per-subtree: a SELECT with one OPTIONAL keeps native BGP legs but the LeftJoin, its filter, and everything above run generically with full Value materialization. Benchmark evidence of the gap: in ThemeQueryCatalog.java 90 of 104 queries contain OPTIONAL, 28 UNION, 22 MINUS, 30 VALUES, 49 BIND; 87 are COUNT-topped and therefore served by the aggregate compiler — but the non-aggregate 'denormalized view' queries (indices 11/12, e.g. ThemeQueryCatalog.java:170-205: nested OPTIONALs + SELECT DISTINCT * + ORDER BY, 199,461 rows) are commented out of the benchmark @Param (ThemeQueryBenchmark.java:118-119), i.e. the exact shapes with no native path are currently unmeasured.

**Recommendation.** Add a row-emitting QueryEvaluationStep that wraps the aggregate compiler's SlotPlan/RowCursor tree and emits FastRowBindingSetView rows (essentially NativeGroupIteration.evaluateAll minus the grouping HashMap), and dispatch to it from LmdbNativeEvaluationStrategy.precompile (LmdbNativeEvaluationStrategy.java:46-59) for non-Group roots: Join/LeftJoin/Union/Difference/Filter/Extension/BindingSetAssignment trees. This reuses ~all of the 3445-line compileTuple machinery and retires the restrictive Builder.collect path. Then re-enable benchmark query indices 11/12 to measure it.

**Verifier refinement.** The recommendation stands but needs three precision fixes: (1) A row-emitting wrapper must call compileTuple with duplicateInsensitive=false and seed requiredAggregateNames with ALL projected variables — the duplicate-insensitive shortcut at LmdbNativeAggregateCompiler.java:296-299 silently drops an OPTIONAL's right arg when its bindings aren't "required", which is only sound under COUNT(DISTINCT)/DISTINCT semantics; reusing it naively for plain SELECT would produce wrong multiplicities and missing optional bindings (duplicateInsensitive=true remains valid only under a Distinct/Reduced root). (2) "Unlocks BIND" overstates Extension support: compileTuple's Extension branch (:360-385) bails for any ExtensionElem whose expr is not a Var or constant (:370-371), so only variable-alias/constant BIND compiles natively today — the catalog's BIND(?x AS ?y) forms qualify, but general expression BIND still needs new work. (3) Minor cite fixes: JoinQueryEvaluationStep is at core/queryalgebra/evaluation/.../impl/evaluationsteps/JoinQueryEvaluationStep.java:44-45; VALUES appears in 29 of 104 catalog queries, not 30; the denormalized-view query spans ThemeQueryCatalog.java:168-205. Additionally, retiring Builder.collect wholesale is premature: LmdbNativeQueryCompiler compiles complex ValueExprs via a lazy BindingSet view fallback (compileFilter returning null aborts the whole plan there, whereas compileBoolean:1018-1024 has a generic predicate fallback), so the aggregate-side path is a strict superset for boolean filters but should be diffed for filter coverage before deleting the old path.

### [CRITICAL][coverage-gaps] FILTER(bound(?x)) / FILTER(!bound(?x)) over a pattern-bound variable returns wrong results natively: SlotBoundFilter reports requiredMask()==0 and is evaluated at depth 0 before any pattern binds the slot
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeQueryCompiler.java:822

**Problem.** compileIdFilter maps Bound to SlotBoundFilter when the var has a slot (LmdbNativeQueryCompiler.java:281-287). SlotBoundFilter.requiredMask() returns 0L (:821-824), so assignFilters computes (required & ~simulated)==0 immediately and pins it at targetDepth 0 (:1197-1200). acceptFilters(0) runs once in findNext right after initialize (:1065), when row.slots[slot] is still UNKNOWN (Arrays.fill with UNKNOWN at :1114) and the base binding set does not contain the name — accept() returns false (:827-829) and the whole iteration closes empty. Concretely, 'SELECT * { ?x ?p ?o FILTER(bound(?x)) }' compiles natively (Bound is an accepted id-filter, the fragment is a plain BGP) and yields zero rows instead of all rows; the dual 'FILTER(!bound(?x))' wraps it in NativeFilter.not (:277-279) and yields ALL rows instead of none. ExternalBoundFilter (:839-842) has the same requiredMask()==0 shape but is only chosen when the var has no slot, which is consistent with base-only semantics — the bug is specifically the slot case.

**Recommendation.** Make SlotBoundFilter.requiredMask() return 1L << slot so assignFilters places it at the depth where the slot is produced; better, constant-fold at compile time: inside a conjunctive BGP a variable that appears in any statement pattern is always bound at full depth, so bound(?slotVar) folds to TRUE and !bound(?slotVar) to FALSE (empty plan). Add a Routine A failing test first (e.g. in the LMDB SPARQL compliance/testsuite path) since this is behavior-changing.

The codebase already proves cursor reuse works: prepareSpoExistenceCursor caches a thread-local cursor renewed across txns (TripleStore.java:684) — but only for fully-bound SPO existence probes.

**Recommendation.** Pool cursors per (dbi, txn) at the Frame/JoinCursor level: keep the LmdbRecordIterator (or a re-seekable variant) alive across probes of the same pattern and reposition with MDB_SET_RANGE on the new key instead of open/close — the renew/reposition code path already exists (:174-199). Hoist index selection to compile time (the bound-mask per join depth is known once order is chosen, so the winning index is fixed per frame). Hold one dataset read stamp per frame lifetime rather than per statements() call, and batch the txn-lock into the seek rather than every next().

**Verifier refinement.** Every binding-producing (partially-bound) INLJ probe pays: per-probe index-selection loop over all indexes (TripleStore.getTriples, TripleStore.java:495-507), a fresh LmdbRecordIterator allocation with mdb_cursor_open under txnLockManager.readLock (LmdbRecordIterator.java:131-152) and mdb_cursor_close on exhaustion (:298-299), plus a nativeSourceLock read stamp and a NativeSourceReadLockedRecordIterator wrapper allocation per statements() call (LmdbSailStore.java:1741-1757). Locking is two acquisitions per probe open plus one txnLockManager.readLock per next() call (:156-162) — not "two locks per row". Fully-bound probes are already exempt: the compilers route them to existence checks (LmdbNativeQueryCompiler.java:626-647) using the cached thread-local SpoExistenceCursor (TripleStore.java:684), and fully-bound statements() probes use mdb_get without a cursor (LmdbRecordIterator.java:141-142, 248-278). Recommended fixes stand: (1) keep a re-seekable LmdbRecordIterator per Frame/JoinCursor alive across probes and reposition via MDB_SET_RANGE (the renew/reposition machinery at :174-199 shows the pattern, though it currently only fires on txn-version change and reuse would need reset of minKeyBuf/fetchNext/closed state); (2) hoist index selection to compile time — getPatternScore depends only on the bound mask, which is fixed per frame depth once join order is chosen (per-frame caching of the winning TripleIndex suffices); (3) hold one nativeSourceLock stamp per frame lifetime instead of per statements() call; (4) amortize txnLockManager.readLock across a batch of next() calls or fold it into seek. Caveat: the aggregate path may go through CompositeNativeLmdbQuerySource/ConcatenatingRecordIterator over multiple sources, so pooling must be per (source, pattern). The 1.5-3x estimate is plausible for probe-dominated queries but unverified.

### [HIGH][join-strategy] No merge/zipper join: sorted-index forward-walk exists only for fully-bound SPO existence checks
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/TripleStore.java:715

**Problem.** LMDB range scans return rows in index key order, so the outer stream of an INLJ over e.g. POSC (bound predicate) is sorted by subject; when the inner pattern's chosen index is also keyed by the join variable first, the probe keys are monotonically non-decreasing and a single forward-walking cursor (merge/zig-zag join) replaces N independent re-seeks. The branch already implements exactly this for one narrow case: spoTripleExistsAnyContext detects monotonic probes via canWalkSpoCursor (TripleStore.java:715, used at :627) and walks MDB_NEXT comparing a cached prefix instead of re-seeking — but only for fully-bound s,p,o existence probes reached through hasTriples (:542-549). Scan probes (PatternCursor.open → source.statements, LmdbNativeQueryCompiler.java:591-614; PatternPlan.open, LmdbNativeAggregateCompiler.java:1678-1704) always create a fresh iterator that re-seeks from scratch (LmdbRecordIterator.java:205-208 MDB_SET_RANGE per iterator). No operator anywhere zips two sorted cursors.

**Recommendation.** Two increments: (1) generalize the walking-existence-cursor trick to binding-producing probes — keep the inner frame's cursor open across outer rows and use MDB_SET_RANGE from the current position when the new probe key is >= the last one (detectable at runtime by comparing keys, or statically when the outer pattern's index order sorts the join slot); (2) for star joins on one variable (?s p1 ?a . ?s p2 ?b with both predicates bound), implement a true zipper on the two POSC ranges that leapfrogs to max(current subjects), which also generalizes to the WCOJ case below.

**Verifier refinement.** The sorted-forward-walk exists only in spoTripleExistsAnyContext and is even narrower than claimed: it fires only for fully-bound s,p,o with UNBOUND context (bound-context probes use exactTripleExists/mdb_get, TripleStore.java:543-545), only when an 'spoc' index is configured (spoContextWildcardIndex, :546-549, :771-779), only outside write transactions (write-txn path opens a throwaway local cursor, :616-621, so canWalkSpoCursor gets a null cachedCursor), and only while subj+pred repeat with non-decreasing obj (:715-720). All binding-producing probes (LmdbNativeQueryCompiler.java:1078-1081 -> :591-614; LmdbNativeAggregateCompiler.java:1678-1704, :2144) allocate a fresh LmdbRecordIterator with a fresh mdb_cursor_open + MDB_SET_RANGE (LmdbRecordIterator.java:144-148, 205-208). No merge/zipper operator exists in the native engine, and the upstream RDF4J merge-join path is also unreachable because LmdbSailStore.getSupportedOrders returns Set.of() and ordered getStatements throws UnsupportedOperationException (LmdbSailStore.java:1936-1944). Recommended increments stand; the existing SpoExistenceCursor (thread-local, mdb_cursor_renew on txn change, cached-prefix compare, TripleStore.java:684-720, 728-748) is a ready-made template for (1); note the generalized version must also handle write-txn cursors (cannot renew) and cursor invalidation on txn version change, and pattern reordering means outer-order sortedness of the join slot must be established from the outer pattern's chosen index field sequence at open time or detected at runtime by key comparison.

### [HIGH][join-strategy] No hash join or inner-side materialization/memoization: unselective, uncorrelated, and repeated probes rescan LMDB per outer row
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeAggregateCompiler.java:2144

**Problem.** The only join algorithm in both native executors is index nested loop: JoinCursor re-opens the right subplan per left row (LmdbNativeAggregateCompiler.java:2135-2145), LeftJoinCursor likewise (:2198+), the BGP Frame re-opens its cursor on every descent (LmdbNativeQueryCompiler.java:1078-1081). Consequences with code evidence: (a) when the right side shares no bound slot (cross product, or a pattern whose only selective term is a constant), the identical LMDB range is fully rescanned once per left row — nothing caches an uncorrelated subplan's result; (b) MinusCursor.hasCompatibleRight opens the right subplan per left row (:2464-2471) with no memoization keyed on the shared-slot values, so identical shared-var combinations re-probe LMDB — O(left × right-probe); (c) ExistsFilter.accept opens its subplan per probed row (:2637-2643) with no memo, even though the sibling ValueSetFilter demonstrates the memo pattern (LongBooleanMemo, :2652, :2677-2694) and CachedCompareFilter too (:2717); (d) when neither side is selective (UNION-of-edges shapes like HC q1/q9), INLJ degenerates to left-cardinality × B-tree-seek instead of one scan + O(1) hash probes. The primitives needed already exist in-file: LongHashSet (:3372+) and LongBooleanMemo (:3206+).

**Recommendation.** Add two cheap physical alternatives chosen at open() time using the existing boundScore/estimate: (1) materialize-and-hash — when the right child has boundScore 0 (uncorrelated) or its estimate is small relative to the left, scan it once into a primitive hash table keyed on the shared slot IDs (LongObjectHashMap of long[] rows, or LongHashSet for semi-joins) and probe; (2) memoize correlated existence — give ExistsFilter and MinusCursor a LongBooleanMemo keyed on the packed shared-slot values (single slot: the id; multiple: mix like GroupKey), mirroring ValueSetFilter.

**Verifier refinement.** Core claim confirmed: both native executors are INLJ-only with no hash join, no inner-side materialization, and no join/EXISTS/MINUS memoization (JoinCursor LmdbNativeAggregateCompiler.java:2135-2145, LeftJoinCursor :2198-2221, MinusCursor :2464-2471, ExistsFilter :2637-2643, BGP Frame LmdbNativeQueryCompiler.java:1078-1081), while LongBooleanMemo (:3206), LongHashSet (:3372) and GroupKey mixing (:3344) already exist in-file. Corrections to the details: (1) uncorrelated MINUS is NOT a problem — SlotPlan.minus drops the right side entirely when sharedMask==0 (:1560-1569) and hasCompatibleRight short-circuits when no shared slot is bound (:2465-2467); the memo win applies only to correlated MINUS (e.g. MEDICAL q9, correlated on ?enc), keyed on the bound shared-slot values. (2) HC q8's FILTER EXISTS is a single statement pattern and is already compiled to StatementPatternExistsFilter (:795-813, :2570-2627) doing one source.has() point probe per row (LmdbSailStore.java:1774) — no subplan reopen; the fix there is a memo keyed on the packed (endId,nodeId) pair, not materialization. ExistsFilter subplan-reopen cost applies to multi-pattern EXISTS such as PHARMA q2 (correlated on ?drug) and to compileFactorizedLeftJoinFilter output (:850). (3) Do not use boundScore==0 to detect uncorrelated right children: composite plans (JoinPlan/UnionPlan/LeftJoinPlan/FilterPlan) all inherit the default boundScore()==0 (:1488-1490). Detect correlation at open() time via (right.producedMask() & row.boundMask()) == 0, which is sound because every variable slot a native plan consumes also appears in its producedMask. (4) Scope note: this machinery only executes Group/aggregate queries (tryCompile :94-104) plus pure BGPs in LmdbNativeQueryCompiler (:113-131); non-aggregate MINUS/EXISTS/UNION queries fall back to StrictEvaluationStrategy, so all cited benchmark shapes (aggregates) are affected, but the recommended operators should land in the SlotPlan layer where they benefit the aggregate path first.

### [HIGH][materialization-boundary] xsd:dateTime/date ID-inlining is effectively dead: millisecond field encoded from getMinute(), no-timezone values throw, and date round-trip emits dateTime lexical form
File: core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/inlined/Dates.java:92

**Problem.** Dates.encodeToLong has `int milli = calendar.getMinute();` (Dates.java:92) instead of getMillisecond(), so the packed millis field holds the minute. Additionally: (a) timezone-less values (the most common lexical form) have getTimezone()==FIELD_UNDEFINED (Integer.MIN_VALUE); tzOffsetStep=/15 fails the range check at Dates.java:122 and throws, so they can never inline; (b) unpackDate (Dates.java:72-75) calls toXMLFormat() on a calendar with hour/minute/second/millis all set by decodeFromLong (:156-165), producing a dateTime-shaped label for an xsd:date, so plain dates also fail the round-trip. Net effect: T_DATETIME/T_DATETIMESTAMP/T_DATE ids (ValueIds.java:41-43) essentially never occur, and every getId() call on a dateTime literal (bulk load, query-compile constant resolution) wastes a full XMLGregorianCalendar construction + toXMLFormat + equals before hitting the dictionary path.

**Recommendation.** Fix line 92 to `calendar.getMillisecond()` (handling FIELD_UNDEFINED as 0 with a distinct 'no-fraction' encoding), add an explicit encodable representation for FIELD_UNDEFINED timezone instead of letting /15 blow the range check, and make unpackDate render only the date portion so the round-trip in ValueStore.getId:1514-1521 passes. Add a unit test asserting ValueIds.isInlined(getId(vf.createLiteral(calendar))) for tz'd, tz-less, fractional and date-only values. Note existing stores are unaffected on disk since the round-trip check meant these ids were never persisted.

**Verifier refinement.** Confirmed as stated, with three corrections/additions: (a) tz-less values do not surface an exception — the IAE from Dates.java:122 is caught in packDateTime/packDate (Dates.java:36/46/56, returning 0L) and again at ValueStore.java:1522; the observable effect is silent dictionary fallback, and for tz-less values the wasted work is calendarValue() parsing + partial encode (the unpack/toXMLFormat/equals cost is only paid for tz'd values). (b) Also fix Dates.java:45: packDateTimeStamp uses ValueIds.T_DATETIME instead of T_DATETIMESTAMP, which would keep dateTimeStamp inlining dead (datatype mismatch in the equals round-trip) even after the milli/tz/date fixes. (c) When fixing, distinguish millisecond==FIELD_UNDEFINED (no fractional second) from millisecond==0 in the 10-bit milli field (e.g. reserve value 1000 or a flag bit for 'no fraction'), and likewise reserve one of the unused tzBits values (e.g. 0x7F) for FIELD_UNDEFINED timezone, so that "10:30:00", "10:30:00Z" and "10:30:00.000Z" round-trip to distinct, label-exact lexical forms — otherwise the equals gate at ValueStore.java:1518 will still reject the common no-fraction/no-tz forms. The proposed unit test should also cover xsd:dateTimeStamp and assert ValueIds.getIdType() matches the expected T_* constant.

# Design: Factorized, Block-at-a-Time Native Engine for LMDB (evolving `LmdbNativeQueryCompiler`)

Verified against current code: `RowState` is a single `long[] slots` + trail (LmdbNativeQueryCompiler.java:1272-1333), the driver is tuple-at-a-time backtracking DFS (`findNext`, :1060-1111) re-opening one `PatternCursor` per probe (`NativePattern.open`, :591-615 → `NativeLmdbQuerySource.statements`, NativeLmdbQuerySource.java:38), `LmdbRecordIterator.next()` returns one reused `long[4]` per call and takes a `StampedLongAdderLockManager` read lock **per tuple** (LmdbRecordIterator.java:156-246), plus a second per-call lock in `LmdbSailDataset.statements` (`NativeSourceReadLockedRecordIterator`, LmdbSailStore.java ~1741). Aggregation materializes all groups in `HashMap<GroupKey, AggState>` (LmdbNativeAggregateCompiler.java:1387-1420).

---

## 1. Intermediate representation: `IdChunk` / `IdVector`

The Kuzu analogue, adapted to "everything is a 64-bit LMDB value ID" (no per-vector logical types needed — one primitive kind):

```java
// final, no interfaces, no polymorphism in the inner loop
final class IdVector {
    final long[] ids;              // capacity CHUNK_CAPACITY = 1024
    // no null mask needed: UNKNOWN_ID / NULL_CONTEXT_ID are in-band sentinels
    IdChunk chunk;                 // back-pointer to shared state
}

final class IdChunk {              // == Kuzu DataChunk + DataChunkState fused
    static final int CHUNK_CAPACITY = 1024;
    final IdVector[] vectors;      // positionally aligned columns
    final int[] slotOfVector;      // vector i binds global slot slotOfVector[i]
    int size;                      // #physical tuples
    int[] sel;                     // selection vector; null == identity 0..size-1
    int selSize;                   // valid when sel != null
    int curIdx = -1;               // -1 => UNFLAT; >=0 => FLAT at this position
    boolean isFlat() { return curIdx >= 0; }
}

final class ChunkSet {             // == Kuzu ResultSet; the factorized tuple
    final IdChunk[] chunks;        // logical tuple = cross product of chunks
    final int[] chunkOfSlot;       // slot -> chunk index (-1 = external/base binding)
    final int[] vecOfSlot;         // slot -> vector index inside chunk
    final long[] constSlots;       // slots seeded from base bindings stay scalar
}
```

**Mapping from the current slot model.** Keep the compiler's slot assignment (`slot()`, :410-419) and `long` masks unchanged — they become the planner's *factorization-group* assignment: at plan time each `NativePattern` that introduces ≥1 new slot gets its own `IdChunk` (its `producedMask ∩ ¬boundBeforeMask`); patterns introducing nothing (existence checks) get none. `chunkOfSlot` replaces `RowState.slots` indexing. `RowBindingSetView` survives unchanged as the *exit* format: a flat snapshot is one position picked from every chunk, copied into a `long[slots]` exactly as `RowState.snapshot()` does today (:1328-1332) — so everything downstream of the engine (generic operators, `getLazyValue` materialization) is untouched.

**JIT rules.** All classes `final`; the driver is one big method over `IdChunk[]` arrays, not an operator-iterator interface chain — the current engine already got this right (`findNext` is monomorphic); preserve it. Filters are the exception: keep the existing `NativeFilter` dispatch but move it out of the per-tuple loop (§2c). Selection vectors are `int[]` reused per chunk (cleared by setting `sel = null`), never reallocated. Do **not** introduce a `ValueVector` interface with flat/unflat subclasses — use the `curIdx` field test; a predictable branch beats bimorphic dispatch.

**Flat/unflat semantics.** Chunk *k* FLAT at `curIdx=i` means "the single tuple at i"; UNFLAT means "all `selSize` tuples". A factorized intermediate for `?a :owns ?b . ?b :name ?n` after scanning `?a` and extending: chunk0 = {a} FLAT, chunk1 = {b, n} UNFLAT (one adjacency batch). Multiplicity of a logical tuple = product of `selSize` over UNFLAT chunks not contributing output vectors — exactly Kuzu's `AggregateInput.multiplicityChunks`.

## 2. Operator redesign

Replace the `Frame[]` DFS with a compiled array of ops executed by one interpreter loop; each op reads/writes `ChunkSet`:

### a) `ScanOp` (leaf)
Fills chunk0's vectors directly from a batched LMDB scan (§4): `int fill = source.statementsBatch(s,p,o,c, out /*4 column long[1024]s*/)`. Column-selects at fill time: only columns whose `Term.hasSlot()` are copied out (the `keyToQuadMatchStatus` decode already produces all four; writing 2 columns instead of 4 is free).

### b) `ExtendOp` — the multi-match probe, replacing per-row `PatternCursor`
This is the core change. Today: for each row, open cursor → iterate → `bind` → recurse (:1078-1093). New:

```java
final class ExtendOp {
    final NativePattern pattern;
    final int inChunk;              // chunk holding the probe key slots
    final int outChunk;             // new chunk for produced slots
    // state:
    int probePos;                   // current position within inChunk selection
    BatchCursor cursor;             // batched LMDB iterator for current key (§4)

    /** Advance: flatten inChunk to next probe position, fill outChunk with
     *  up to CHUNK_CAPACITY matches. Returns false when inChunk exhausted. */
    boolean nextBatch(ChunkSet cs) throws IOException {
        for (;;) {
            if (cursor == null) {
                if (!advanceProbe(cs)) return false;      // sets inChunk FLAT at probePos
                cursor = pattern.openBatch(source, cs);   // keys from flat positions
            }
            int n = cursor.fillChunk(cs.chunks[outChunk]); // one JNI-amortized batch
            if (n > 0) { cs.chunks[outChunk].size = n; cs.chunks[outChunk].curIdx = -1; return true; }
            cursor.close(); cursor = null;                 // dangling probe: no rows
        }
    }
}
```

Semantics: `ExtendOp` **flattens its input chunk** (one probe key at a time — required, since LMDB gives one sorted range per key) but **emits an unflat chunk**, so multiplicities multiply *additively in storage*: a 2-hop `(a)→{b₁..bₖ}→{f₁..fₖ}` costs 2k+1 stored IDs, not k². This is the factorization-creating step, identical to Kuzu's Extend. Note the current `bind()` per-position consistency checks (repeated variable in one pattern, constant mismatch, named-graph null-context rejection, :650-665) become vectorized post-fill filters producing a selection vector on `outChunk`.

**When results stay factorized vs flatten.** Flattening is implicit and lazy: chunk *j* is flattened only when a *later* `ExtendOp` or filter needs one of its slots as a probe key / cross-chunk operand — the op iterates `curIdx` over the selection (Kuzu's `Flatten`). The plan orders patterns exactly as `chooseOrder` does today; the factorization win materializes automatically for **suffix patterns whose slots nobody downstream probes** — the tail of the join order stays unflat. Rule for the planner: after computing the order, walk backwards; chunk *j* stays unflat iff no op at depth > j consumes a slot of chunk *j* as input. Projection/DISTINCT/aggregation (§3) then exploit it. When the consumer is the generic RDF4J layer (`NativePlanIteration` clients pulling `BindingSet`s), the final enumerator is a `FlatTupleIterator` doing odometer-style cross-product enumeration over unflat chunks into the existing `RowBindingSetView` — same output as today, but the *internal* work (filters, existence checks, aggregates) already happened factorized.

### c) Filters over ID vectors
Native ID filters (`SameTerm`, type tests, constant EQ/NE — :264-341) become vector kernels: `int filterChunk(IdChunk c, int slotVec)` writing a selection vector, one virtual call per **1024 tuples** instead of per tuple. The type-test kernels are pure bit ops on the ID (`ValueIds.getIdType`) — perfect SuperWord candidates over `long[]`. `GenericNativeFilter` (:792-810) evaluates per *distinct flat combination*: flatten only the chunks in its `requiredMask`, evaluate once, apply the verdict to whole unflat chunks when the filter doesn't touch them — this alone is a factorized win for the ubiquitous `OPTIONAL{} FILTER(?optVar…)` benchmark shape. Filter placement (`assignFilters`, :1192-1218) is unchanged, just depth → op index. Fix in passing: `SlotBoundFilter.requiredMask()==0` bug (:821-829) — required mask must include the tested slot.

## 3. Aggregation over factorized intermediates (`LmdbNativeAggregateCompiler`)

Current: one `GroupKey` allocation + `HashMap` probe per fully-enumerated row (:1406-1414). New contract: the child plan yields `ChunkSet` states, and `AggState.add(RowState)` becomes:

```java
void addFactorized(ChunkSet cs, int aggSlot) {
    long mult = 1;                              // product of unflat, non-arg chunk sizes
    for (IdChunk ch : cs.chunks)
        if (!ch.isFlat() && ch != chunkOf(aggSlot)) mult *= ch.selSize;
    // COUNT(?x): if ?x's chunk is unflat, add selSize * mult without touching ids[]
    counts[i] += vectorLen(cs, aggSlot) * mult;
}
```

- **COUNT(?x)** never enumerates: `k` neighbors × `m` other-branch tuples = `k*m` added in O(#chunks). This directly attacks HIGHLY_CONNECTED q9 (40k groups) and the LIBRARY q8 timeout class.
- **COUNT(DISTINCT ?x)**: only the chunk containing `?x` must be walked — insert its ≤1024 ids into the existing `LongHashSet` (:3372-3444) *once per group per batch*, ignoring multiplicity (idempotent). Other chunks contribute nothing. No flatten.
- **GROUP BY**: group-key slots must be flat when a batch is committed — the planner emits flatten of key chunks only (Kuzu's `FlattenAll` on key groups). Common fast case: keys live in the *first* (probe-side) chunks which are flat anyway during extension; the unflat tail is pure aggregate payload. Replace `HashMap<GroupKey,AggState>` with an open-addressing `long[]`-keyed table (keys inlined as `groupSlots.length` consecutive longs, murmur-mixed like `LongHashSet`) — kills the per-row boxed `GroupKey` allocation independent of factorization.
- Future SUM/MIN/MAX slot in cleanly: SUM(x) = Σ x·mult over the x-chunk; MIN/MAX per distinct value of the x-chunk.

## 4. Feeding vectors from LMDB: batch fill per JNI crossing

New method on the storage layer (TripleStore + `NativeLmdbQuerySource`):

```java
/** Fill up to cap quads; returns count; 0 = exhausted. One lock acquire,
 *  one cursor position, up to cap mdb_cursor_get(MDB_NEXT) calls per invocation. */
int nextBatch(long[] subjOut, long[] predOut, long[] objOut, long[] ctxOut,
              int cap, int colMask);
```

Implementation inside `LmdbRecordIterator`: hoist the `txnLockManager.readLock()` / version check / `NativeSourceReadLockedRecordIterator` lock **outside** the while-loop — amortized 1024×; the loop body is the existing `mdb_cursor_get(MDB_NEXT)` + `keyToQuadMatchStatus` decode writing into column arrays at index `n` instead of the shared `quad`. `colMask` skips columns without slots. Each `mdb_cursor_get` is one JNI call regardless (LMDB has no multi-record fetch primitive short of `MDB_MULTIPLE`/DUPFIXED, which the key layout here can't use), so the win is *not* fewer JNI crossings per se but: per-tuple lock elimination (currently 2 lock round-trips/row), no per-row `long[]`-handoff megamorphism, decode loop that C2 can unroll, and elimination of the per-probe `LmdbRecordIterator` allocation via a **per-`ExtendOp` reusable cursor**: keep one `mdb_cursor` open per op for the plan's lifetime, reposition with `MDB_SET_RANGE` per probe key (mirroring the existing thread-local SPO-existence cursor trick, TripleStore.java:684-715, generalized). Since probe keys within an `ExtendOp` arrive in ascending slot-ID order whenever the *input* chunk came from a sorted scan, the `canWalkSpoCursor`-style forward-walk (MDB_NEXT + prefix compare instead of re-seek) generalizes into a poor-man's merge join — this is the RDF-3X SIP hook and should be part of the batch cursor from day one.

## 5. Migration path (each step independently benchmarkable via `ThemeQueryBenchmark` + `scripts/run-single-benchmark.sh`)

1. **Batch cursor, tuple engine unchanged (small, deliberate first step).** Add `RecordIterator.nextBatch(...)` with a default single-record shim; implement natively in `LmdbRecordIterator` (lock hoisting + reusable arrays). `PatternCursor` internally buffers a batch and dribbles tuples to the unchanged DFS. Pure Routine-B-shaped storage change; expect wins on scan-heavy queries (HC q9, LIBRARY q0) from lock amortization alone.
2. **Per-op persistent cursors**: pool one LMDB cursor per `Frame` (reposition instead of open/close on every backtrack re-entry, :1078-1095). Independent of chunks.
3. **Leaf + last-hop chunking**: introduce `IdChunk`/`ChunkSet`; convert only the *deepest* pattern of the join order to `ExtendOp` emitting an unflat chunk, with a `FlatTupleIterator` at the top. Benchmarks isolate factorized-tail benefit (star queries, COUNT-topped plans).
4. **Factorized COUNT/COUNT DISTINCT** in `LmdbNativeAggregateCompiler`: multiplicity-product `AggState.addFactorized` + flat open-addressing group table. Targets LIBRARY q8 timeout, HC q1/q9, PHARMA q2.
5. **Full pipeline chunking**: all patterns become `ScanOp`/`ExtendOp`; vectorized native filters with selection vectors; `GenericNativeFilter` evaluated per-flat-combination.
6. **Existential skip**: projected-away / DISTINCT-irrelevant unflat chunks degrade `ExtendOp` to first-match (`has()`-style) probes — semi-join semantics for free from factorization metadata.
7. **SIP/merge extension**: ascending-key forward-walk cursors (step 4 of storage), then bitmap semi-join filters between plan segments. WCOJ (intersect op for cyclic cores, SOCIAL q8 triangle) only after this, reusing sorted batch cursors as leapfrog iterators.

Steps 1–2 don't touch the compiler; 3–4 are behavior-neutral w.r.t. results (Routine B gates likely fail on scope → Routine A/D with plan-equivalence tests; the benchmark harness's count verification is the correctness backstop).

## 6. What NOT to copy from Kuzu

- **Morsel-driven parallelism / thread-per-pipeline**: RDF4J's iteration contract is single-consumer pull; LMDB read txns are thread-confined here (thread-local txn in `TxnManager`). Parallelism belongs above the engine, later.
- **FactorizedTable + hash-join build sides**: the engine is INLJ over always-available B+tree permutations; there is no build pipeline to materialize. Skip ASP-Join's accumulate phase; adopt only its semi-mask idea (step 7).
- **2048 capacity + overflow AuxiliaryBuffers**: IDs are fixed 8 bytes; no string/list overflow machinery. 1024 `long`s = 8KB/vector keeps a whole `ChunkSet` L1/L2-resident; tune empirically.
- **Null masks & logical types**: in-band sentinels (`UNKNOWN_ID`, `NULL_CONTEXT_ID`) suffice; a per-vector `NullMask` adds branches for nothing.
- **Planner-level flatten resolver as a separate optimizer phase**: with ≤60 slots and single-`long` masks, flatten decisions are a 20-line backward walk over the chosen order at `initialize()` time — keep the runtime ordering model rather than Kuzu's static `f_group_pos` schema, since the engine re-orders per `evaluate(bindings)` (:1138) when invoked correlated.
- **Codegen**: no Umbra-style compilation; C2 on final classes + array loops is the strategy (per the vectorized-vs-compiled tradeoff, and JVM code-cache/deopt costs).
- **Roaring bitmaps for semi-masks**: LMDB IDs are not dense node offsets; start with `LongHashSet`/sorted `long[]` masks.

Key files: LmdbNativeQueryCompiler.java (:591-750 cursors, :1060-1333 driver/row), LmdbNativeAggregateCompiler.java (:1387-1477 aggregation), LmdbRecordIterator.java (:156-246 hot loop), LmdbSailStore.java (~1741 `statements()` lock), NativeLmdbQuerySource.java (SPI to extend with `nextBatch`), TripleStore.java (:684-715 walkable-cursor precedent).

### Kuzu factorized execution
# Kuzu Factorized Query Processing — Engineering Notes

## 1. ValueVector / DataChunk: the factorized tuple representation

Kuzu is a vectorized, block-based processor, but unlike DuckDB an operator's working set (`ResultSet`) is a **set of DataChunks**, and the intermediate relation is interpreted as the **Cartesian product of the tuples in each chunk** ("vector groups" in the CIDR 2023 paper; an *f-representation* in Olteanu's factorization theory).

Concrete structures (master branch, `src/include/common/`):

- `ValueVector` (`vector/value_vector.h`): `LogicalType dataType`; `std::unique_ptr<uint8_t[]> valueBuffer` (fixed-width slots, `numBytesPerValue`); `NullMask nullMask`; `std::unique_ptr<AuxiliaryBuffer> auxiliaryBuffer` (strings/lists/structs overflow); `std::shared_ptr<DataChunkState> state`. Capacity is either 1 or `DEFAULT_VECTOR_CAPACITY` = `1 << KUZU_VECTOR_CAPACITY_LOG2`, default **2^11 = 2048** (`cmake/templates/system_config.h.in`, `CMakeLists.txt`).
- `DataChunk` (`data_chunk/data_chunk.h`): `std::vector<std::shared_ptr<ValueVector>> valueVectors` + one shared `DataChunkState`. All vectors in a chunk are positionally aligned (same length, same selection).
- `DataChunkState` (`data_chunk/data_chunk_state.h`): `std::shared_ptr<SelectionVector> selVector` and `FStateType fStateType` where `enum class FStateType : uint8_t { FLAT, UNFLAT }` — "F stands for Factorization". `isFlat()/setToFlat()/setToUnflat()`; `getSingleValueDataChunkState()` builds a state for single-value vectors.
- `SelectionVector` (`data_chunk/sel_vector.h`): `selectedPositions` pointer + `selectedSize`, with STATIC mode (identity/"unfiltered": contiguous positions from 0, no materialized indices) and DYNAMIC mode (`setToFiltered()`, mutable `sel_t` buffer holding arbitrary positions). Filters never copy data; they shrink/redirect the selection.

**Flat vs unflat semantics.** In the CIDR paper each vector group carries `curIdx`: *flat* if `curIdx ≥ 0` (the group denotes the single tuple at position `curIdx`), *unflat* if `curIdx = -1` (the group denotes all `selectedSize` tuples). In current code `curIdx` is realized as: `fStateType == FLAT` + a selection vector of size 1 pointing at the current position (the `Flatten` operator overwrites the sel vector, below). A factorized intermediate `{(a.ID,a.owner) flat} × {(t.ID,b.ID) unflat} × {(c.ID) unflat}` is exactly three DataChunks; a chunk pair of 2048-vectors can thus represent up to 2048×2048 = 4M logical tuples. The 2-hop example: `(v1,Karim) × {v(1,1,b)..v(1,k,b)} × {v(1,1,f)..v(1,k,f)}` encodes k² tuples in 2k+1 values; the blog's Liz example stores 20,000 flat tuples as 400 values.

**Multiplicity.** Projection can drop an unflat chunk while its cardinality must survive: after projecting out unflat `b` from `(a.owner=Karim) × {b.ID=v(1,1..k)}`, the tuple `(Karim)` has multiplicity k. Operators that count (aggregates, inserts) must not assume 1 tuple per factorized tuple; they multiply by the selected sizes of in-scope chunks that no longer contribute vectors (see §6).

**FactorizedTable** (`processor/result/factorized_table.h`) is the materialized twin used by sinks (hash join build, ASP-Join accumulate, ORDER BY, aggregation): `FactorizedTableSchema` holds per-column `ColumnSchema` with an **isUnflat flag** and `dataChunkPos` (which factorization group the column belongs to); flat columns are fixed-width fields at `colOffset` inside fixed-size tuples in `flatTupleBlockCollection`; unflat columns store an 8-byte `overflow_value_t` pointing into `unFlatTupleBlockCollection` heap blocks (an overflow list = one factor). `append(vectors)` branches on `vector.state->isFlat()`; readers are `readFlatCol`/`readUnflatCol` (the latter re-inflates an overflow list into an unflat ValueVector, restoring factorization on output); `FlatTupleIterator` enumerates the cross product only when a client demands flat tuples.

## 2. Operators, planner schema, and where flattening happens

The planner tracks factorization statically: `planner::Schema` assigns every expression to an `f_group_pos` (factorization group); groups map 1:1 to DataChunks at runtime. Flattening is planned, not ad hoc — `flatten_resolver.h` provides `FlattenAll` and `FlattenAllButOne::getGroupsPosToFlatten(exprs, schema)`, and the planner inserts `LogicalFlatten` accordingly.

- **Scans produce factorization**: `ScanNodeTable` emits an unflat node-ID/property chunk; `Extend` (rel-table scan) requires the bound node **flat** and writes its adjacency list (nbr IDs + rel properties) into a *new unflat chunk* — this is the factorization-creating step for every m-n hop.
- **Flatten** (`processor/operator/flatten.h`): iterates an unflat chunk one position per `getNextTuplesInternal()` call: `FlattenLocalState{currentIdx, sizeToFlatten}`; it inherits `SelVectorOverWriter` and `resetCurrentSelVector()` installs a size-1 selection at `currentIdx`, setting the state FLAT. Cost: multiplies upstream iterations, so the optimizer minimizes flattens (its DP cost metric = number of factorized tuples; it keeps the best plan per factorization structure per subquery).
- **HashJoin build side**: join keys must be effectively flat; the planner flattens key groups below `HashJoinBuild` (`keyState` in `hash_join_build.h` is "set to any flat key state if all keys are flat"), and payload chunks may stay unflat — they are appended into the FactorizedTable as overflow lists, preserving their factors. **Probe side stays factorized**: probe key must be flat (or the plan uses ASP-Join, §3), and matched build payloads are emitted as unflat vectors (`readUnflatCol`), so factorization flows through the join.
- **Expression evaluation**: binary evaluators support exactly two operand shapes — (i) both operands in the same group (classic vectorized loop over shared sel vector), or (ii) operands in different groups, in which case plan enumeration guarantees at least one side is flat with size 1 (broadcast loop). Two unflat operands from different groups are illegal → `FlattenAllButOne`. `FlattenAll` is used where an operator needs a fully flat row (e.g., all-groups-dependent expressions, distinct hashing).

## 3. S-Join, ASP-Join, sideways information passing

Kuzu's second design goal is *sequential-only scans*; SIP reconciles that with factorization:

- **S-Join**: vanilla 2-pipeline hash join + semijoin filter. Pipeline 1 builds the hash table on flat key `a.ID` and simultaneously builds a bitmask over node offsets (bit i ⇒ a.ID=i was hashed); the mask is passed sideways to the probe pipeline's scan, which sequentially scans only passing `a.owner` values. Requires the **build-side key flat** in a good factorization.
- **ASP-Join** (accumulate–semijoin–probe, 3 pipelines) handles the case where the natural probe key (e.g. `b.ID` after an Extend) is **unflat** and flattening would destroy factorization: (1) *Probe Accumulation*: materialize probe-side factorized tuples into FactorizedTable FT (sequential writes) and build a semijoin filter from the unflat `b.ID` values; (2) *Hash Table Build*: build side scans (e.g. `b.owner`) consult the filter, scanning only needed tuples sequentially; (3) *Probe*: re-scan FT in factorized form and probe. Overhead = one materialization; benefit = factorization kept + no random I/O.
- Runtime artifacts: `SemiMasker` operator (`semi_masker.h`) with `SemiMaskerSharedState::masksPerTable` (per-table `SemiMask`, thread-local masks merged via `mergeToGlobal()`), masks keyed by `nodeID.offset` per `tableID`; implementation `Roaring32/64BitmapSemiMask` (`common/roaring_mask.h`); table scans check the mask per node group/offset. Planner rule: node variable `a_i` may pass SIP if it carries a predicate or joins an edge.

## 4. Worst-case-optimal joins

Cyclic subqueries get at least one enumerated plan whose top is **multiway ASP-Join** (Generic-Join-style: extend a prefix tuple to a new variable c by intersecting ℓ adjacency lists). Pipelines: (1) accumulate prefix FT + ℓ semijoin filters (one per participating edge relation; filters over the same label/direction are unioned into one hash table); (2..ℓ+1) scan each adjacency list passing the filter, **sort it, and hash it** into a per-relation hash table keyed on the bound variable; (ℓ+2) rescan FT, `Flatten` each bound variable (unflat group of k → k flat iterations), probe the ℓ tables (`Intersect` operator, `intersect.h`: `probeHTs` collects `probedFlatTuples` from `sharedHTs`; `intersectLists` intersects sorted lists smallest-first via `intersectSelVectors`), output c as a **new unflat vector** — WCOJ output is itself factorized. Triangle benchmark: 1.62 s (WCO) vs 51.17 s (binary joins) on 7.6M edges.

## 5. Morsel-driven parallelism

Per the paper: "We adopt morsel-driven parallelism [Leis et al.]. Parallel tasks run copies of the same pipeline and coordinate to get morsels of node IDs/properties to scan until no morsels remain." Implementation: each pipeline → a `Task` with `maxNumThreads`; `TaskScheduler` (`common/task_system/task_scheduler.h`) keeps a FIFO deque, workers `getTaskAndRegister()` and clone the pipeline's operator tree; source operators hand out morsels from shared state (table-function morsels default to `DEFAULT_VECTOR_CAPACITY`; node-table scans partition by node group, `NODE_GROUP_SIZE = 2^17 = 131072` rows). Sinks (FactorizedTables, semi masks, aggregate tables) have thread-local states merged into shared state at task end.

## 6. Aggregates and ORDER BY/LIMIT without enumeration

- **COUNT/SUM**: aggregate updates take a `multiplicity` argument. `AggregateInput` (`aggregate/aggregate_input.h`) = `{ValueVector* aggregateVector; std::vector<DataChunk*> multiplicityChunks}` — the in-scope chunks *not* containing the aggregated vector; multiplicity = ∏ their `state->getSelSize()`. `CountFunction::updatePos` is literally `state->count += multiplicity`. So `COUNT(*)` over `(a) × {b₁..b₁₀₀}` inspects the unflat size (100) instead of enumerating; SUM(x) adds `x * multiplicity`.
- **MIN/MAX**: evaluated per distinct value of the factor containing the argument — 100 comparisons instead of 10K in the blog's example (idempotent aggregates ignore multiplicity).
- **COUNT(DISTINCT)/aggregates needing flat rows**: planner emits `FlattenAll` on the relevant groups (distinct hashing keys must be flat); `containDistinctAggregate()` routes to a distinct-collection hash table.
- **ORDER BY/LIMIT**: `OrderBy` is a sink over FactorizedTables — key columns are planned flat and radix-encoded (`OrderByKeyEncoder`) for merge sort, but **payload columns may remain unflat** overflow lists; sorted scan re-emits payload factors as unflat vectors, and `TopK` keeps only heads, so the cross product is never fully enumerated unless the client iterates it (`FlatTupleIterator`).

### QLever internals
# QLever Internals: Mechanisms for Performance

## 1. ID-based query processing (`ValueId`)

- Every RDF term is a 64-bit `ValueId`: **4 most-significant bits = datatype tag, 60 bits = payload** (`src/global/ValueId.h`). Datatypes: `Undefined, Bool, Int, Double, VocabIndex, LocalVocabIndex, TextRecordIndex, WordVocabIndex, BlankNodeIndex, Date, GeoPoint, EncodedVal`.
- Payload encodings: ints are 60-bit two's complement (range ±2^59); doubles are IEEE-754 truncated to 49 mantissa bits; dates are bit-packed (year/month/day/hour/min/sec/tz shifted into the payload) so bitwise `<` equals chronological `<`; GeoPoint packs lat/long. `VocabIndex` payloads are positions in the on-disk vocabulary, which is **sorted**, so ID order = lexicographic term order.
- Consequence: triples, joins, ORDER BY and most FILTER comparisons operate on raw `uint64_t` comparisons — **no dictionary lookup on the hot path**. Because the type tag sits in the high bits, sorted ID runs are automatically partitioned by datatype; range predicates (`?x >= 60 && ?x <= 65`) become ID-interval checks.
- Query-time constants not in the vocabulary (e.g. `"Bob"` in a FILTER) get a `LocalVocabIndex`; comparison against vocab IDs uses the constant's *insertion-position range* (lower/upper Vocab-ID bound) in the sorted vocabulary, so even cross-vocabulary comparisons stay integer-based. `LocalVocabIndex` cleverly stores an aligned pointer whose 4 low bits are zero, reusing them for the tag.
- Vocabulary is split internal/external: long literals (>~50 chars, <1% of Freebase literals but 50% of the bytes) are externalized to disk and only materialized at result-export time. The on-disk vocabulary file is `items ... offsets`, giving random access to item *i* with two seeks.

## 2. Index layout: permutations + compressed blocks

- Six permutations **PSO, POS, SPO, SOP, OSP, OPS**; PSO+POS suffice if no predicate variables occur (user-selectable). Each permutation is a giant sorted list of ID triples stored as a sequence of **compressed blocks, column-oriented**: each block holds N triples, each column compressed independently (ZSTD) so a two-column scan decompresses only two columns. Block size is configurable (`blocksizePermutationPerColumn_`, a per-column byte budget); small relations sharing a `col0Id` are packed into shared blocks, large relations span many blocks.
- **`CompressedBlockMetadata`** (kept in RAM, one entry per block): per-column file offset + compressed size, `numRows_`, the **full first and last triple of the block (inclusive)**, optional list of contained graph IDs (or `nullopt` if too many), and a flag for graph-only duplicates. First/last triples make the value range of a block known **without decompression** — the basis for all block skipping (scans, prefilters, join pushdown).
- A scan (`ScanSpecification`: 0–3 fixed columns) binary-searches block metadata by first/last triple to find candidate blocks, decompresses only those, and trims boundary blocks. Permutations carry extra payload columns: a graph column, and SPO/OSP carry two **pattern columns** (pattern ID of subject/object) supporting the "pattern trick" (§7).

## 3. Join algorithms

All in `src/util/JoinAlgorithms/JoinAlgorithms.h`; `Join` requires sorted inputs (planner inserts `Sort` when needed):

- **Zipper/merge join** (`zipperJoinWithUndef`): linear-time merge over the shared sort order; UNDEF-aware variant treats rows compatible if each position is equal or one side UNDEF (UNDEF rows are sorted to the front and handled by a separate range). Reports output sortedness for downstream reuse.
- **Galloping join** (`gallopingJoin`): when one side is ≥`GALLOP_THRESHOLD` (=1000)× larger and UNDEF-free — linear over the small side, exponential+binary search in the large side.
- **Block-based lazy joins**: `zipperJoinForBlocksWithoutUndef` merges two *streams of blocks*, buffering only blocks whose ranges overlap (tracked via min of last elements) and computing Cartesian products across block boundaries; a `WithPotentialUndef` variant covers single-column UNDEF, and `specialOptionalJoinForBlocks` handles OPTIONAL with UNDEF only in the last join column.
- **Join pushdown into scans**: joining two `IndexScan`s (`computeResultForTwoIndexScans`) mutually prefilters block metadata — each scan reads only blocks whose [first,last] range intersects the other's; joining a materialized `IdTable` with an `IndexScan` (`computeResultForIndexScanAndIdTable`) binary-searches the scan's block metadata by the table's join-column values and fetches only matching blocks (RDF-3X-style SIP, but via metadata). Hash join exists (`hashJoinImpl`) only as a fallback for unsorted/lazy corner cases; the engine strongly prefers order-exploiting joins.

## 4. FILTER prefiltering on block metadata

- FILTER expressions are compiled into **`PrefilterExpression`s** evaluated against `CompressedBlockMetadata` *before* the INDEX SCAN decompresses anything: a block is skipped iff its [first,last] triple range provably contains no satisfying value (Baumann 2025 thesis; `PrefilterExpressionIndex`).
- Supported: relational `< <= > >= = !=` against constants, `isIri/isBlank/isLiteral/isNumeric` (datatype-tag ranges!), prefix filters (`STRSTARTS`, anchored `REGEX("^...")` → vocab-ID interval), `IN`/`NOT IN`, composed with `&&`/`||`/`!` (merged/intersected block sets), plus composites like `YEAR(?d)=2000` (date payload range). Works because IDs order by type-then-value, so every such predicate maps to unions of ID intervals comparable against block bounds. Planner attaches the prefilter to the scan whose sorted first free column is the filtered variable (POS is key for object-value filters).

## 5. Lazy streaming results and LIMIT

- Operations produce a `Result` that is either fully materialized or a **generator of `IdTable` chunks** (`lazyScan()` yields blocks via `asyncParallelBlockGenerator` — parallel decompression feeding a queue). Scans, joins, FILTER, GROUP BY (sorted case), DISTINCT (sorted case) and export are chunk-streaming; consumers pull, so **LIMIT terminates upstream computation** after enough chunks — top-k over huge scans never materializes the scan. IndexScan supports LIMIT/OFFSET natively.
- Cache integration (Textor-Falconi 2025): lazy results can be cached while streaming; if a stream exceeds the configurable memory budget it is either passed through uncached or aborted per limits, avoiding the old OFFSET/LIMIT chunking workaround. Fully materialized subresults are cached and reused across queries (subtree cache keyed on canonical execution trees).

## 6. Query planning

- Build a graph: node per triple pattern, edge per shared variable. **Seed** the DP table with all viable index-scan plans per triple (each useful permutation = one candidate). **DP over connected subgraphs**: row *k* holds all plans covering *k* nodes; merge rows *i+j=k* when disjoint and connected; every merge inserts Join (+Sort if the join column isn't the plan's sort order). **Prune**: keep only the cheapest plan per equivalence class (same triples covered, same filters applied, same result ordering) — ordering is part of plan identity, so "interesting orders" survive.
- Costs count elements touched; size estimates use per-column **multiplicities**: join size `s = α·m_a·m_b·min(d_a,d_b)` with distinct-count propagation (`d_i = min(d_orig, s'_orig)`); scan multiplicities are precomputed in metadata. Filters applied greedily as soon as variables are bound (both filtered/unfiltered candidates kept in DP mode). Cached subtrees get cost ~0. Above a subgraph-count threshold the planner switches from DP to **Greedy Operator Ordering**.

## 7. Large GROUP BY / DISTINCT

- Default: sort-based streaming aggregation (group boundaries in sorted chunks; `LazyGroupBy` state machine commits groups across chunk boundaries — constant memory).
- **Hash-map optimization** (runtime flag): replaces Sort+scan for AVG/COUNT/MIN/MAX/SUM/GROUP_CONCAT/SAMPLE (non-DISTINCT), accumulating per-group aggregate state keyed on group columns, templated on #group-columns.
- **Metadata-only GROUP BY**: `COUNT` over a full 3-var scan with no GROUP BY → answer from permutation metadata; `GROUP BY ?p (COUNT)` over full scan → `getDistinctCol0IdsAndCounts()` precomputed at build time; join-with-full-scan counts via `getResultSizeOfScan()` per key — no scan at all.
- **Pattern trick**: per-subject predicate-sets ("patterns") precomputed at indexing, stored as extra ID columns in SPO/OSP, make `SELECT ?p (COUNT(DISTINCT ?s))`-style queries run over patterns instead of all triples.
- DISTINCT on sorted results is a streaming adjacent-dedup; ORDER BY/GROUP BY prefer plans already sorted (planner keeps sort-order-distinct plans alive).

### Factorization / WCOJ theory
# Factorized Query Processing & WCOJ for a Graph/RDF Engine — Implementation Guidance

## 1. Factorized representations (Olteanu & Závodný)

**Core idea:** represent a join result as an algebraic expression over singletons using union and product, exploiting distributivity: `⋃ a×(⋃ b)×(⋃ c)` instead of the flat Cartesian expansion. Two systems:

- **f-representations**: tree-shaped factorization guided by a *variable order* (d-tree) compatible with the query. Size bound: `Θ(N^s(Q))` where `s(Q)` is the maximum over the d-tree of fractional edge cover numbers of induced subqueries.
- **d-representations**: f-representations plus subexpression sharing (DAG). Size bound: `O(N^fhtw(Q))` — fractional hypertree width. Since `fhtw(Q) ≤ ρ*(Q)` (the AGM exponent for flat results) and `fhtw = 1` for acyclic (α-acyclic) queries, **any acyclic BGP has a linear-size factorized result** even when the flat result is exponentially larger. Many-to-many hubs in RDF (e.g., `?a :knows ?b . ?b :knows ?c`) are exactly the case where flat intermediates explode but factorized ones stay linear.

**FDB** (the Oxford engine) evaluates factorized-in/factorized-out with constant-delay enumeration of tuples from the representation. **Actionable:** you don't need a full FDB; the pragmatic runtime encoding is Kuzu's: a "factorized tuple" is a Cartesian product of *vector groups*, where each group is either **flat** (single value, one tuple) or **unflat** (a vector of up to ~2048 values). One multiplicity per group; join keys live in flat groups; dangling many-side matches stay unflat. This drops multiplication of intermediate sizes to addition for star/path patterns.

## 2. Aggregation, DISTINCT, ORDER BY over factorized results

- **COUNT/SUM/aggregates**: factorization turns products of unions into products of *partial aggregates* — `COUNT` over a factorized subtree is the product of branch counts, computed bottom-up in one pass. Formalized by **FAQ** (Abo Khamis, Ngo, Rudra — InsideOut algorithm: semiring aggregation via variable elimination, `O(N^faqw)` where faqw generalizes fhtw) and **AJAR** (Joglekar, Puttagunta, Ré — GHDs with aggregate pushdown over annotated relations). The crucial constraint in both: **free (group-by) variables must sit above aggregated variables in the variable order/decomposition**; the optimizer must search only "aggregate-compatible" orders. Bakibayev et al. (VLDB'13) show group-by aggregates over factorized joins beat RDBMSs by orders of magnitude because partial aggregation collapses subtrees to scalars early.
- **DISTINCT / projection**: in a d-representation, projecting away variables = deleting subtrees below the projected-out variables, then deduplicating unions — cheap if projected-out variables are *leaves/lower* in the d-tree. `SELECT DISTINCT ?x WHERE { ?x :p ?y }` should never enumerate `?y`: emit `?x` once per non-empty subtree (existential semantics). This is the single highest-value factorization trick for SPARQL: implement "check-nonempty and skip" (effectively semi-join evaluation) for variables that are projected away or only DISTINCT-irrelevant.
- **ORDER BY**: enumeration in sorted order is free **iff the sort attributes form a prefix of (a path in) the variable order**. Then you can stream results in order with no sort, and `ORDER BY … LIMIT k` becomes top-k enumeration without materialization. If not a prefix, restructure the d-tree at plan time (swap operator) or flatten just the offending groups. Actionable: make the optimizer prefer variable orders whose prefix matches ORDER BY/GROUP BY keys; RDF index orderings (SPO/POS/…) often supply them for free.

## 3. Worst-case optimal joins in practice

**Algorithms:** Leapfrog Triejoin (LogicBlox) and Generic Join (NPRR) run in `O(N^ρ*(Q))` — the AGM bound — by resolving *one variable at a time* via k-way intersection of sorted/trie iterators. LFTJ needs trie access in a fixed attribute order per relation. **RDF stores with all index permutations (SPO, POS, OSP, …) get LFTJ essentially for free** — your LMDB B+trees already provide sorted `seek(≥v)` iterators; leapfrogging is just alternating gallops.

**Practical systems and hybrid planning:**
- **Umbra** (Freitag et al., VLDB'20): builds **hash-trie** indexes on the fly (no precomputed tries), and its optimizer keeps binary-join plans by default, routing only the subplan through WCOJ **when estimated intermediate cardinality exceeds the worst-case output bound** (growing, cyclic, skewed parts). This is the reference design for retrofitting WCOJ into an existing binary-join optimizer.
- **Graphflow/Kuzu** (Mhedhbi & Salihoglu, VLDB'19): plans mix binary joins with *extend/intersect* (multiway intersection) operators, costed with a novel **intersection-cost (i-cost)** metric plus adaptive runtime reordering of vertex orderings. Kuzu's WCOJ operator emits **unflat vectors** (the intersection result as one factorized group) — WCOJ and factorization compose naturally.
- **EmptyHeaded**: GHD-based plans + adaptive set layouts (bitset for dense, sorted arrays for sparse) with SIMD intersections; aggregate pushdown over GHDs.
- **Free Join** (Wang et al., SIGMOD'23): unifies both — start from an optimized binary plan, convert to a Free Join plan, use **COLT (column-oriented lazy trie)** built lazily only for probed prefixes, with vectorized batch probing. Best current blueprint if you want one operator framework instead of two.

**Guidance:** use WCOJ **only for cyclic subpatterns** (triangles, diamonds, cliques — common in social/knowledge graphs); acyclic remainders (stars, paths — the majority of SPARQL BGPs) are faster with binary/merge joins + factorization. Detect cyclicity of the BGP join graph at plan time; carve out cyclic cores, evaluate with LFTJ over your existing index permutations, join the (factorized) result to the acyclic rest with binary joins.

## 4. Vectorized vs compiled execution on the JVM

Kersten et al. (VLDB'18): the paradigms are within ~2× of each other overall — **codegen wins on computation-heavy, tight per-tuple logic** (expression-dense, index-nested-loop, LFTJ-style seek loops); **vectorization wins on memory-bound scans**, offers better adaptivity/profiling, and has zero compile latency. WCOJ is control-flow-heavy and per-tuple, which is why Umbra compiles it; Kuzu instead reshaped it into vector-at-a-time multiway intersection producing unflat vectors.

**For a JVM engine (RDF4J):**
- Prefer **vectorized (block-at-a-time) execution with primitive `long[]` dictionary-ID vectors + selection vectors** (DuckDB/Kuzu style, batch 1024–2048). This sidesteps the JVM's codegen weaknesses: C2 *is* your compiler, but only if call sites stay mono/bimorphic. RDF4J's current `CloseableIteration` chains are megamorphic virtual dispatch per tuple — the vectorized model amortizes dispatch to once per 2048 tuples, which is most of the codegen benefit without generating bytecode.
- Avoid full Spark-style whole-stage codegen (Janino): compile latency, code-cache pressure, deopt storms, and debugging cost are high; reserve **small-scope bytecode/MethodHandle generation for expression/filter evaluation only** (Trino's model) if profiling shows it matters.
- Keep hot loops allocation-free, arrays-of-primitives (no boxed `Value` in the pipeline — resolve to IDs early, materialize terms late), and consider `jdk.incubator.vector` for SIMD sorted-set intersection/galloping in the WCOJ leaf.
- Watch profile pollution: reuse of generic operator classes across very different plans makes C2 profiles polymorphic; per-operator-instance dispatch tables or a small number of specialized operator variants help.

## 5. Sideways information passing / semi-join reduction

- **RDF-3X-style SIP** (Neumann & Weikum): merge joins over sorted index scans exchange "next possible key" hints so scans can skip ahead — directly applicable to LMDB cursors (`seek(≥hint)`), nearly free to implement, large wins on selective joins.
- **LIP / Bloom-filter passing** (Zhu & Patel, VLDB'17; standard in DuckDB/Umbra): build-side of each hash join publishes a Bloom filter (or, with dense dictionary IDs, a plain **bitmap** — cheaper and exact) that is pushed into probe-side scans. With dictionary-encoded RDF, bitmaps over ID ranges beat Bloom filters.
- **Kuzu's ASP-join** (accumulate → semijoin filter → probe): fully accumulate the build side first, pass a node-ID mask to probe-side scans so entire adjacency/column files are skipped. Composes with factorization since the mask filters flat (key) groups only.
- SIP is your robustness hedge for join-order mistakes: it makes bad orders cheap rather than requiring perfect cardinality estimation, and it is the cheapest of all five techniques to retrofit.

**Priority order for implementation value:** (1) ID-vectorized block execution; (2) SIP hints on merge/index joins + bitmap semi-join filters on hash joins; (3) factorized intermediate groups for many-to-many hubs + existential skip for DISTINCT/projection; (4) partial-aggregate pushdown through factorized groups; (5) LFTJ for cyclic cores over existing index permutations with Umbra/Graphflow-style hybrid costing.

## 7. Subsystem architecture notes

### native-compiler
## Purpose

New in commit f2c7a022f7 ("wip native engine"); does not exist on develop. `LmdbNativeQueryCompiler` (core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeQueryCompiler.java) compiles LMDB-local BGP+Filter fragments into a slot-oriented executor operating purely on primitive `long` value IDs, bypassing the generic `StrictEvaluationStrategy` join machinery and Value materialization. Entry point: `LmdbNativeEvaluationStrategy.precompile` (LmdbNativeEvaluationStrategy.java:46-60) — aggregate compiler gets first shot, then this, else generic. Gated by `rdf4j.lmdb.nativeQueryEngine.enabled` (default true, LmdbNativeEvaluationStrategy.java:42) and a usable `NativeLmdbQuerySource` (extracted at :62-69; `CompositeNativeLmdbQuerySource.combine` returns null if sources have mismatched `idSpace`).

## Mechanics

1. **Collect** (`Builder.collect`, :112-136): recursively flattens `Filter` (conjuncts split, :430-438), `Join`, `StatementPattern`, `SingletonSet`. Any other node → `false` → caller falls back.
2. **Slot assignment** (`slot`, :410-419): first-seen order, name→int; slot IDs double as bit positions in `long` masks, capped at `MAX_NATIVE_SLOTS=60` (:77, checked :146). Constant terms are resolved to IDs at compile time (`compileTerm`, :225-244); dictionary-miss constant → pattern impossible → `NativePlan.empty()` (zero results, :139-140).
3. **Context/graph handling** (`compileContextConstraint`, :184-223): dataset default/named graphs become a `long[]` of context IDs; `RDF4J.NIL`/`SESAME.NIL` map to `NULL_CONTEXT_ID=0`; unknown graph IDs dropped; all-unknown → empty plan. `Scope.NAMED_CONTEXTS` with null context is rejected at open (:600-601) and per-quad in `bind` (:651-653). Fixed multi-context sets iterate one LMDB range per context lazily (`PatternCursor.contexts`, :708-737).
4. **Ordering** — done per `evaluate()` in `NativePlanIteration.initialize` (:1113-1140): incoming bindings seed slots (native-ID fast path when base is a `RowBindingSetView` sharing the same idSpace, :1119-1121; unknown external value → impossible, :1128-1132). `chooseOrder` (:1142-1171) greedily picks the pattern with the highest bound score (constant=2, bound slot=1 per position; `Term.boundScore`, :506-514), tie-broken by static cardinality estimate computed at compile time via `tripleStore.cardinality` (:179, LmdbSailStore.java:1785-1795).
5. **Filter placement** (`assignFilters`, :1192-1218): each filter attaches to the earliest depth where its `requiredMask` is covered by bound slots.
6. **INLJ driver** (`findNext`, :1060-1111): iterative backtracking DFS over `Frame[]`. Each depth opens a `PatternCursor` (`NativePattern.open`, :591-614) with currently bound slots as LMDB lookup keys. Fully-bound, non-producing patterns become `mdb_get` existence checks (`openAsExistenceCheck`, :626-648; named-scope + unknown context falls back to scanning, :642-646). `bind` writes quad IDs into `RowState.slots` with a trail for O(changed) rollback (:1295-1316). Filters at depth+1 run per binding; passing rows advance; exhausted frames close, backtrack, `release` rolls the trail (:1255-1260).
7. **Emission**: at full depth, `row.snapshot()` copies the slot array into an immutable `RowBindingSetView` (:1328-1332), a lazy `BindingSet` that materializes values only on `getValue` via `source.lazyValue(id)` (:1459-1465 → `ValueStore.getLazyValue`, LmdbSailStore.java:1723-1733); slot ID 0 renders as null (null context).
8. **Filter compilation** (`compileFilter`, :246-262): native ID filters (`compileIdFilter`, :264-341) cover And/Or/Not composition, `Bound`, `SameTerm` (raw ID equality, :850-870), `Compare` EQ/NE only when one side is an IRI/BNode constant (SPARQL value-equality ≠ ID-equality for literals, comment :297-301), and type tests (IsURI/IsResource/IsLiteral/IsBNode/IsTriple/IsNumeric) decoded from ID type bits via `ValueIds.getIdType` (:898-919). Everything else wraps `strategy.precompile(expr).asPredicate()` over the mutable `RowState.bindingView` (`GenericNativeFilter`, :792-810) — stays in the native pipeline but materializes values lazily on access.

## Perf-relevant design decisions

- Rows are one `long[]` (:1289); filter/pattern dependencies are single-`long` bitmasks (:571-584, :945-947).
- `LmdbRecordIterator.next()` returns a **reused** `long[4]` quad buffer (LmdbRecordIterator.java:75, 97, 229) — zero allocation per tuple; `bind` copies into slots immediately.
- Trail-based undo avoids copying the slot array per depth (:1279-1316).
- One LMDB read txn per dataset (LmdbSailStore.java:1684-1693); cursor iteration guarded by a `StampedLongAdderLockManager` read-lock **per `next()` call** (LmdbRecordIterator.java:156-162) plus a txn-version check (:174) — per-tuple locking overhead on the hottest path.
- Existence checks use `mdb_get` (`has`, LmdbSailStore.java:1774-1782) instead of cursors.
- `getBindingNames` cached only on stable snapshots (:1377-1395).

## Weaknesses/opportunities

- **`Bound` correctness risk**: `SlotBoundFilter.requiredMask()==0` (:822-824) so `assignFilters` places it at depth 0; for `FILTER(BOUND(?x))` where ?x is bound by a pattern (not the base), the slot is still UNKNOWN at depth 0 → `accept` false (:827-829) → entire query yields empty. Same for `ExternalBoundFilter` (:840-842) interacting with `Not`.
- Join order uses only **static, compile-time estimates** with unbound slots (:179, :1159); `source.count` (NativeLmdbQuerySource.java:40) is never used for dynamic re-estimation once slots are bound — deep INLJ chains can pick bad orders on skewed data.
- `chooseOrder` (O(n²)) + `assignFilters` (ArrayList allocations, :1193-1216) + `Frame[]`/`RowState` construction run **per `evaluate(bindings)` call** (:989-1021) — expensive when the plan is the inner side of a generic join invoked per outer row.
- `RowBindingSetView.slot()` is a linear string-equals scan (:1446-1453); every `getValue`/`hasBinding` from generic filters or downstream operators is O(#slots).
- Per-result allocation: slot-array copy + view (:1328-1332); `iterator()` builds ArrayDeque + SimpleBinding per binding and calls `base.hasBinding` per slot (:1357-1374, :1422-1430).
- No caching of materialized `Value`s per slot in `RowState`; generic filters at multiple depths re-call `lazyValue` for the same ID.
- Cursors are closed and re-created on every backtrack re-entry (:1078-1080, :1095) — no cursor/iterator pooling at frame level.
- `Compare` EQ between two slot variables always falls to the value-based evaluator (:302-303) even when both are known resources at runtime.
- `valueId`/`valueIdAllowUnknown` are identical (:421-427) — the intended "allow unknown" semantics (compile-time miss for filter operands should be `never()`, not plan-impossible) only works by accident.
- Multi-context estimates ignore the restriction (`staticContextForEstimate` only for singleton sets, :535-537).

## Fallbacks (returns null → generic RDF4J evaluation)

- Any algebra node other than Filter/Join/StatementPattern/SingletonSet in the fragment (:135).
- No statement patterns collected (pure SingletonSet/filter fragments, :142-145).
- More than 60 distinct variables (:146-148).
- Strategy-level: property disabled, no LMDB `NativeLmdbQuerySource` (non-LMDB triple source or composite idSpace mismatch, CompositeNativeLmdbQuerySource.java:30-44), or `!hasStatementsInSource()` (LmdbNativeEvaluationStrategy.java:47).
- Not fallbacks but empty-result short-circuits: constant not in dictionary (:238), dataset graphs all unknown (:219-220), type-invalid constant per field (:447-457), external base binding not in dictionary (:1128-1132).

### aggregate-compiler
## Purpose

`LmdbNativeAggregateCompiler.java` (3445 lines, new on `optimize-lmdb`, absent from `develop`; commits `f2c7a022f7`, `a3e06e33c6`) extends the branch's native slot/ID engine from BGPs to aggregation. It intercepts `Group` / `Filter(Group)` (HAVING) algebra nodes and evaluates the *entire child algebra* as a primitive-`long` ID row program, materializing RDF `Value`s only for final aggregate rows (class doc :76-83). It supports far more algebra than the sibling `LmdbNativeQueryCompiler` (which only flattens Filter/Join/StatementPattern/SingletonSet conjunctions, `LmdbNativeQueryCompiler.java:113-137`): Join, LeftJoin, Union, Difference (MINUS), Filter, Extension (var-copy BINDs only), VALUES, SingletonSet (:262-390).

**Query shapes handled:** GROUP BY on any var set (:253-260) incl. global aggregation (:1392-1404); **COUNT and COUNT(DISTINCT) only**, over a plain `Var` or constant (:227-251). No SUM/AVG/MIN/MAX/SAMPLE/GROUP_CONCAT, and **no COUNT(\*)** (null arg fails `instanceof Var` :235-238 → whole compile bails). HAVING is compiled as a generic predicate post-filter over the native group rows (:122-129). ORDER BY / LIMIT / DISTINCT-projection are not handled here; those wrappers stay generic and reach this compiler when the strategy recursively precompiles their `Group` child.

## Mechanics

1. **Hook:** `LmdbNativeEvaluationStrategy.precompile` tries this compiler first, then the BGP compiler, else generic (`LmdbNativeEvaluationStrategy.java:47-59`); gated on `rdf4j.lmdb.nativeQueryEngine.enabled` and a store-backed `NativeLmdbQuerySource` (:42, :62-69).
2. **Compile:** aggregates → group slots → duplicate-insensitivity analysis (:135-156). If all aggregates are DISTINCT, or a `_anon_having_` count has a HAVING lower bound ≥1 (:172-225), duplicates don't matter, unlocking rewrites: drop LeftJoins whose right side binds nothing required (:296-299); fold `Join(data, Filter(VALUES))` into per-constant pattern alternatives (`MultiValuePatternPlan`, :560-600, :439-467); rewrite null-rejecting `Filter(LeftJoin)` into an EXISTS semi-join (:834-851); push `?x = c || ?x IN (...)` into a statement pattern (:392-409).
3. **Row model:** vars → int slots capped at `MAX_NATIVE_SLOTS = 60` (:89, :152) so plans can use 64-bit `producedMask` bitsets. `RowState` holds `long[] slots` with a trail-based mark/rollback undo log — no row copying (:2779-2836).
4. **Execution:** `NativeGroupIteration.evaluateAll` (:1387-1420) seeds slots from outer bindings — accepting raw native IDs from another `FastRowBindingSetView` in the same `idSpace` (:1430-1436) — then drives one `RowCursor` tree pull-loop; per row it builds a `GroupKey` (long[] copy + cached hash, :3344-3370) into `HashMap<GroupKey, AggState>` (:1406-1414). `AggState` = `long[] counts` + one open-addressing `LongHashSet` per DISTINCT aggregate (:3309-3342, custom murmur-mixed set :3372-3444). Output rows: `source.lazyValue(id)` for keys, `BigInteger` literal for counts (:1460-1476).
5. **Joins:** `MultiJoinPlan` flattens reorderable children and greedily orders at `open()` by boundScore, then non-I/O estimate, using `SYNTHETIC_BOUND` markers (:2055-2097); `JoinCursor` is an index nested loop re-opening the right side per left row (:2122-2160). `PatternPlan.open` turns bound slots into LMDB lookup keys; fully-bound patterns degrade to `source.has()` existence probes (:1691, :1791-1810). Runtime `estimate()` deliberately never touches LMDB (comment :1713-1722); static estimates are captured at compile time via `source.estimate` (:927-955).
6. **Filters:** And/Or/Not composition (:982-996); EXISTS over a single pattern → point-lookup filter (:795-813) else subplan `ExistsFilter` (:2629-2644); IN/= → `ValueSetFilter` with ID fast path + memoized value-equality fallback (:2646-2709); Compare → `CachedCompareFilter` (:2711-2747); anything else falls back to a generic precompiled `Predicate` over the lazy `FastRowBindingSetView` (:1018-1024, :2838-2988) — this keeps complex filters *inside* the native pipeline.

## Perf-relevant design decisions

- All intermediate state is primitive; zero `BindingSet` allocation on the native path; trail undo instead of row cloning (:2802-2825).
- ID equality is only trusted for URI/BNode/Triple IDs (`safeResourceId` :1224-1227); literal comparisons detour through the value-space fallback, memoized per-ID in `LongBooleanMemo` (open-addressing, :3206-3281; unbounded growth).
- Global-aggregate `AggState` pre-sizes each DISTINCT set at 65,536 entries (:1393) → ~1 MB `long[]` per distinct aggregate up-front; grouped case uses 16 (:1410).
- Constant-IN filters compile one `PatternPlan` per constant with per-constant static LMDB estimates (:444-457, :927-955) — compile-time I/O proportional to IN-list size.
- `MultiValuePatternPlan` picks a single alternative when the constrained slot is already runtime-bound, or a fallback scan+filter when correlated (:1869-1896).

## Weaknesses / opportunities

- Only COUNT/COUNT DISTINCT; **COUNT(\*), SUM/AVG/MIN/MAX all bail to generic** (:232-238) — biggest coverage gap.
- Grouping allocates a boxed `GroupKey` + `long[]` per input row into `java.util.HashMap` (:1409) — no primitive/flat hash aggregation; hot allocation on large inputs.
- Results fully materialized (`ArrayList` + map, :1406-1419); no streaming or spill.
- `MinusCursor` re-opens the right subplan per left row (:2450-2471) and recomputes `boundMask()` by scanning all slots (:2827-2835) — O(n·m) MINUS.
- `ExistsFilter` opens its subplan per probed row (:2637-2643); nested `MultiJoinPlan.chooseOrder` is O(k²) with estimate calls per open (:2055-2086), repeated per outer row for correlated plans.
- `FastRowBindingSetView.slot()` does a two-pass linear name scan per lookup (:2963-2975) — per-row cost when generic filter fallback is active.
- **Correctness risk:** `GroupKey` groups on raw IDs (:3353-3359); if an equal literal can carry two encodings (inlined vs stored — the reason `safeResourceId` gates equality elsewhere), groups could split.
- **Edge case:** `AggState.add` counts `NULL_CONTEXT_ID` (0) as a bound value (:3325-3337) while `toBindingSet` suppresses it as a binding (:1465) — possible overcount for COUNT over a context var in the default graph.

## Fallbacks (returns `null` → `LmdbNativeQueryCompiler` → generic `StrictEvaluationStrategy`)

- Root expr not `Group` or `Filter(Group)` (:97-103); native engine disabled or non-LMDB source (`LmdbNativeEvaluationStrategy.java:47`).
- Any non-COUNT aggregate, COUNT(\*), or non-Var COUNT arg (:232-238); constant with unknown store ID (:241-244).
- Unsupported child algebra — subqueries/Projection, Order, Slice, nested Group, property paths, Service, etc. (`compileTuple` default :389).
- BIND with any computed expression (Extension expr not a plain `Var`, :370-372); >60 slots (:152-154); VALUES rows with unknown or non-resource IDs (:970-972); LeftJoin condition not compilable (:307-311).
- Distinct short-circuits (not fallbacks): unmatchable constants in a pattern set `impossible` → empty native plan (:266-272, :148-151); outer binding absent from store → zero-count/empty result (:1437-1448).

### storage
## Purpose

On develop, `TripleStore.getTriples` returned an `LmdbRecordIterator` that scanned an LMDB B-tree range, filtered keys with a heap-allocated `GroupMatcher`, checked range end with a per-key `mdb_cmp` JNI call, and decoded matching keys in a second pass. This branch turns the same layer into the primitive-ID data source for the native slot executor (`LmdbNativeQueryCompiler`/`LmdbNativeAggregateCompiler`), which calls it once per join probe. Changes: (1) single-pass decode+match straight from native key memory via `sun.misc.Unsafe` (no `GroupMatcher`, no per-key `ByteBuffer` wrapper, no per-key `mdb_cmp`); (2) prefix-aware early scan termination; (3) cursor-free `mdb_get` point lookups when all 4 fields are bound; (4) new storage entry points `countTriples` (TripleStore.java:509), `hasTriples(Txn,…)` (TripleStore.java:538) with a thread-local renewable existence cursor; (5) `EmptyRecordIterator` singleton; (6) write-path scratch reuse.

## Mechanics

**Index selection.** `getTriples` inlines the best-index loop (TripleStore.java:495-507): for each of the configured indexes, `TripleIndex.getPatternScore` (TripleIndex.java:165) counts how many *leading* fields of the index's field order (`spoc`, `posc`, …) are bound; highest score wins; `score > 0` enables range search. New `getRangePrefixLength` (TripleIndex.java:207) computes the same count with `subj/pred/obj > 0`, `context >= 0`.

**Range scan setup.** `LmdbRecordIterator` ctor (LmdbRecordIterator.java:91-153): min key written into a pooled 36-byte direct buffer (`Pool.getKeyBuffer`, Pool.java:39). If `prefixLength > 0`, **no max key is built at all** (LmdbRecordIterator.java:109-111) — range end is detected during decode. If `prefixLength == 4`, `exactKeySearch=true` and **no cursor is opened** (`cursor = 0`, line 141-142). Otherwise `mdb_cursor_open` under the txn read lock (lines 144-148). `MDBVal`s and key buffers come from a thread-local freelist `Pool` (Pool.java:22-46, 1024 entries).

**Iteration.** `next()` (LmdbRecordIterator.java:156-246): first call does `MDB_SET_RANGE` on the min key (or `MDB_NEXT` for full scan); subsequent calls do `MDB_NEXT` (`fetchNext` flag). Each key is decoded once by `TripleIndex.keyToQuadMatchStatus(long keyAddress, …)` (TripleIndex.java:370-388): reads varints directly from the LMDB-owned address via `Varint.readUnsigned(address, firstByte)` (Varint.java:415) + `firstToLength` table (Varint.java:440), compares against precomputed `matchSubj/Pred/Obj/Context` sentinels (-1 = wildcard, LmdbRecordIterator.java:93-96), and fills the reused `quad` long[4]. Tri-state result: `KEY_MATCH` → return `quad` (same array every row, zero allocation); `KEY_FILTERED` → `MDB_NEXT`; `KEY_OUT_OF_RANGE` (mismatch inside the bound prefix) → stop scan (LmdbRecordIterator.java:224-238). `mdb_cmp` vs a max key survives only for the `prefixLength==0` corner (line 217, 280-282).

**Exact key.** `nextExactKey` (LmdbRecordIterator.java:248-278): one `mdb_get`, decode from `minKeyBuf`, then auto-close. No cursor lifecycle at all.

**Cursor renew.** On txn version change, `mdb_cursor_renew` + reposition with `MDB_SET`/`MDB_SET_RANGE` on the last returned quad (LmdbRecordIterator.java:174-199) — pre-existing, untested (TODO line 175).

**Close.** `closeInternal` (LmdbRecordIterator.java:284-318): closes cursor (if any), returns MDBVals/buffers to the pool; takes the txn write lock only when closed asynchronously from a non-owner thread. Iterators self-close on exhaustion (lines 241, 276).

**Existence/count.** `hasTriples` (TripleStore.java:538): fully-bound + known context → `exactTripleExists` via `mdb_get` (TripleStore.java:912); fully-bound SPO + wildcard context → `spoTripleExistsAnyContext` (TripleStore.java:591) using a **thread-local cached cursor** (`prepareSpoExistenceCursor`, TripleStore.java:684) that is `mdb_cursor_renew`ed across read txns and, for monotonically increasing object probes (`canWalkSpoCursor`, TripleStore.java:715), *walks forward with `MDB_NEXT`* comparing a cached byte prefix instead of re-seeking — merge-join-style existence checking. `countTriples` (TripleStore.java:509) shortcuts all-wildcard to `mdb_stat` (TripleStore.java:573) and exact-4 to a point get, else counts by full iteration.

## Perf-relevant design decisions

- `LmdbUtil` Unsafe layer (LmdbUtil.java:51-152): `setMDBValData` (line 114) pokes size/pointer straight into the MDBVal struct, bypassing LWJGL checks; `mdbValDataAddress` (line 139) reads the key pointer without creating a `DirectByteBuffer` view — develop's `keyData.mv_data()` allocated a fresh buffer wrapper **per row**.
- Zero-allocation hot loop: reused `quad`, pooled MDBVals/buffers, no matcher, no max-key buffer for prefix scans.
- `EmptyRecordIterator.INSTANCE` for statically-empty sources (LmdbSailStore.java:1747).
- `LmdbStatementIterator` 4-slot MRU ID→Value cache (LmdbStatementIterator.java:39-46, 92-116) for the materializing path.
- Write path: aligned-write scratch arrays/maps reused across batches (TripleStore.java:1516-1531, ensure* at ~1729), `count < subj.length` restriction dropped (TripleStore.java:1513); `fieldSeqString` cached for per-iterator stats naming (TripleIndex.java:53).

## Weaknesses/opportunities

- **Per-probe cursor open/close**: every `statements()` probe in a nested-loop join allocates an `LmdbRecordIterator`, opens and closes an LMDB cursor, and re-runs index selection over all indexes (TripleStore.java:495-507, LmdbRecordIterator.java:144-148). The renewable-cursor trick exists only for SPO existence (TripleStore.java:684); generalizing it to scan iterators is the obvious next win.
- **`countTriples` is O(n)** for any partially-bound pattern (TripleStore.java:527-535) — no prefix-cardinality structure; weak vs QLever for `COUNT(*)` aggregates.
- **Two lock round-trips per row**: `txnLockManager.readLock()` on every `next()` (LmdbRecordIterator.java:159) even for single-row exact lookups.
- **Unsafe on JDK 25+**: `sun.misc.Unsafe` memory-access methods are terminally deprecated; only initialization failures are caught (LmdbUtil.java:66-79) — `unsafe.getByte` throwing at call time (future JDK) is unhandled.
- **Existence-cursor races**: `closeSpoExistenceCursors` (TripleStore.java:783) closes cursors owned by *other threads* from `setIndexes`/`close` (TripleStore.java:261,432,439) without synchronization against concurrent `spoTripleExistsAnyContext`; `ThreadLocal.remove()` only affects the calling thread.
- Renew-after-version-change path still repositions with the stale `txn` handle and has no test coverage (LmdbRecordIterator.java:174-177).
- `nextExactKey` decodes/validates the key it wrote itself (LmdbRecordIterator.java:258-261) — pure overhead; `mdb_get` success already implies a match.

## Fallbacks

- `-Drdf4j.lmdb.nativeQueryEngine.enabled=false` (LmdbNativeEvaluationStrategy.java:44) or triple source not backed by `NativeLmdbQuerySource` datasets (lines 64-71) → `StrictEvaluationStrategy.precompile`.
- Compiler `collect()` rejects any algebra node other than `Filter`/`Join`/`StatementPattern`/`SingletonSet` (LmdbNativeQueryCompiler.java:113-137) → null → generic evaluator; likewise no patterns (line 146-148) or >60 distinct vars (`MAX_NATIVE_SLOTS`, lines 77, 150-152; slot bitmask is a `long`).
- Invalid constant for a field position or constant with no LMDB ID → `impossible` → `NativePlan.empty()` (native empty result, not fallback; LmdbNativeQueryCompiler.java:124-130, 165-170).
- Complex filters do *not* bail: they stay native via `GenericNativeFilter` wrapping the precompiled RDF4J predicate over a lazy ID-backed BindingSet (LmdbNativeQueryCompiler.java:246-262).
- Storage-level: `EmptyRecordIterator` when `!hasStatementsInSource()` (LmdbSailStore.java:1745-1748); aligned bulk writes fall back to `storeTriplesIndividually` on cache/resize/fallback triggers (TripleStore.java:1513-1514, 1535, 1657).

### valuestore
## Purpose

`ValueStore` owns the bidirectional mapping between RDF `Value`s and 64-bit LMDB IDs; the native slot engine works purely in ID space and touches it only at the edges (constant→ID at compile, ID→Value at projection). Branch changes vs develop: (1) ID→Value cache redesigned from direct-mapped to **4-way set-associative** with a bit-spreading hash (ValueStore.java:160-162, 275-279); (2) new **in-memory id→hashCode cache** decoupled from the optional on-disk hash file (ValueStore.java:281-284, 691-720); (3) per-object caches on `LmdbIRI`/`LmdbLiteral` (`cachedHash`, `cachedCalendarValue`, `cachedIntegerValue`); (4) Unsafe-based `MDBVal` struct access in `getData` (ValueStore.java:787-806, LmdbUtil.java:114-157); (5) `TxnManager.doWith` releases the store read lock before closing/pooling the read txn (TxnManager.java:240-259); (6) `ValueStoreRevision.getStoredHash` no longer gated on `valueHashCacheEnabled` (ValueStoreRevision.java:109-115).

## Mechanics

**ID→Value (lazy).** Native rows are `long[] slots` (LmdbNativeQueryCompiler.java:1277-1330). Materialization happens only via `RowBindingSetView.value()` → `source.lazyValue(id)` (LmdbNativeQueryCompiler.java:1459-1465) → LmdbSailStore.java:1723-1731 → `ValueStore.getLazyValue` (ValueStore.java:897-934). That method takes a `revisionLock` read stamp, probes the set-associative cache (`cachedValue`, :815-833), and on miss allocates a **shell** `LmdbIRI`/`LmdbLiteral`/`LmdbBNode`/`LmdbTripleTerm` holding only `(lazyRevision, id)` — no LMDB read. String data resolves on first accessor: `LmdbIRI.init()` (LmdbIRI.java:153-165) → `ValueStoreRevision.Lazy.resolveValue` (ValueStoreRevision.java:93-100) → `ValueStore.resolveValue` (:983-1016): inlined-literal unpack (:985), cache copy with revision-id check (:993), else `getData` → `data2value` → `cacheValue`. `getData` (:787) runs under `ValueStore.readTransaction` (:1414-1439), which reuses the **thread-local LMDB read txn** (`txnManager.getReadTxn()`) under the `StampedLongAdderLockManager` read lock; it writes the key into the `MDBVal` struct via raw `Unsafe.putAddress` (LmdbUtil.setMDBValData:114-128) and copies the result out with `Unsafe.copyMemory` (copyValueBytes, ValueStore.java:797-803) — no `ByteBuffer` view allocation. The store-global lazy-cache probe was deliberately removed from the hot path (comment :900-902); `LmdbStatementIterator` instead keeps a 4-entry per-iterator MRU (LmdbStatementIterator.java:39-46, 92-116).

**Value→ID.** `getId` (:1481-1574): own-value + current-revision fast path (:1486-1491); `valueIDCache` (ConcurrentCache) and `commonVocabulary` CHM (:1497-1500); literal **inlining** with pack/unpack/equals round-trip (:1512-1525); else `value2data` + `findId` (:1541-1543) — `mdb_get` on data→id for keys ≤ MAX_KEY_SIZE, hash-bucket cursor probing for long values (:1176-1276). On hit the ID is memoized into the value and `valueIDCache`, and the RDF hash stored (`storeHashIfAbsent` :1566, 2037).

**Hashing.** `LmdbIRI.hashCode` (LmdbIRI.java:220-247) / `LmdbLiteral.hashCode` (LmdbLiteral.java:317-338): object-local `cachedHash` → `revision.getStoredHash` → `ValueStore.getStoredHash` (:691-720): direct-mapped in-memory `valueHashCache` (:761-764), `pendingHashUpdates`, then on-disk `ValueStoreHashFile`. `storeHash` (:722) populates the memory cache and early-returns when `hashFile == null`, so DISTINCT/GROUP BY hash reuse now works without the hash-file config.

**Txn lifecycle.** `doWith` (TxnManager.java:240-259) now unlocks the read stamp *before* `txn.close()`, shortening the read-lock critical section (close takes `pool`/`active` monitors + `notifyAll`). `Txn.close()` restructured to explicit `synchronized(this)` (:371-388) — semantically equivalent to the old synchronized method.

## Perf-relevant design decisions

- 4-way associativity + `spreadValueId` (:873-880) because encoded IDs share low bits (type tags) and direct mapping thrashed; victim way chosen pseudo-randomly from `id>>>8` (:864).
- Cache arrays are plain `long[]`/`LmdbValue[]` — intentionally racy, validated by re-checking `value.getInternalID() == id` (:817-831).
- Shell values mean projected-but-never-inspected bindings cost one allocation and zero LMDB reads.
- `LmdbLiteral` caches `calendarValue`/`integerValue` (LmdbLiteral.java:236-254), avoiding re-parsing per FILTER comparison; `calendarValue()` still clones defensively per call.
- `setFromInitializedValue` now sets `initialized = true` (LmdbIRI.java:98-99, LmdbLiteral.java:185-189), preventing repeat resolution.
- Constructor allocates caches before `open()` (:272-291).

## Weaknesses/opportunities

1. **Torn-pair race in valueHashCache**: `cacheStoredHash` writes two parallel arrays unsynchronized (:766-777); concurrent writers on a colliding index can pair id A with hash(B); `getCachedStoredHash` (:761) then returns a wrong hash that gets **permanently poisoned** into `LmdbIRI.cachedHash` (LmdbIRI.java:225-227) → equal values with unequal hashCodes → duplicate rows from DISTINCT/GROUP BY. Unlike the value cache, no recheck is possible.
2. `cachedHash`/`getStoredHash` use 0 as sentinel — values whose real hash is 0 recompute forever (LmdbIRI.java:222, ValueStore.java:692).
3. `getLazyValue` takes a StampedLock read stamp per term (:898) — candidate for `tryOptimisticRead` on the hottest path.
4. One shell allocation per cache miss, no interning — high-cardinality projections allocate per row.
5. `resolveValue` runs without `revisionLock` (unlike `getValue`), and has leftover debug comment `// System.out.println(id)` (:1007).
6. `RowBindingSetView.slot()` is a linear `String.equals` scan (LmdbNativeQueryCompiler.java:1446-1453) — O(slots) per named lookup.
7. MRU miss path does 8 field writes (LmdbStatementIterator.java:107-114).
8. `valueHashCache` sized `max(1024, valueCacheSize)` direct-mapped (:282) — silent conflict eviction on large stores.
9. `doWith` closes/pools the txn after dropping the read lock — safe today (`close`/`setActive` both `synchronized(this)` with `closed` guard), but resize/deactivate coupling is subtle and uncommented (TxnManager.java:252-257 vs :409-419).

## Fallbacks

- `LmdbNativeEvaluationStrategy.precompile` (LmdbNativeEvaluationStrategy.java:46-61): native path requires `rdf4j.lmdb.nativeQueryEngine.enabled` (default true), an extractable `NativeLmdbQuerySource` (only `SailDatasetTripleTermSource`, :64-71), and `hasStatementsInSource()`; aggregate compiler tried first, then BGP compiler; either returning `null` falls through to `StrictEvaluationStrategy` (generic evaluation).
- Compilers return `null` for any unsupported algebra node (e.g. LmdbNativeQueryCompiler.java:88, 147, 340, 715).
- Native IDs cross binding-set boundaries only when `idSpace()` (the `ValueStore` instance, LmdbSailStore.java:1737) matches (`sameIdSpace`, LmdbNativeQueryCompiler.java:1442-1444); otherwise values re-enter via `getId`.
- Within the native path: constants unknown to the store yield `UNKNOWN_ID` → empty cursor rather than error; complex filters evaluate through the lazy `RowBindingSetView` (materializing only referenced slots) instead of aborting the plan.

### wiring
## Purpose

On `optimize-lmdb`, LMDB gets a store-native SPARQL fast path: BGP(+filter) fragments and aggregate shapes are compiled to a slot-oriented executor that joins over raw `long` LMDB value IDs, never materializing `Statement`/`Value` objects until rows leave the engine. vs develop: `LmdbEvaluationStrategyFactory` is replaced by `LmdbNativeEvaluationStrategyFactory` (rename, LmdbStore.java:95,189,489-497) which is now the unconditional automatic factory (develop switched between Default/Lmdb factories based on sketch-estimator readiness; LmdbStore.java:184-192 now always returns the native factory unless an explicit factory is set). New plumbing: `NativeLmdbQuerySource` (SPI), `CompositeNativeLmdbQuerySource`, `LmdbNativeEvaluationStrategy`, `LmdbNativeQueryCompiler` (1467 lines), `LmdbNativeAggregateCompiler` (3445 lines), plus base-layer hooks (`SailDatasetTripleTermSource`, `SailDataset.hasStatements`, dataset-wrapper transparency flags) and a generic-engine EXISTS fast path.

## Mechanics

1. **Strategy creation**: `SailSourceConnection.evaluateInternal` builds `SailDatasetTripleTermSource(vf, rdfDataset)` (SailSourceConnection.java:260) and asks the store for a strategy; `LmdbNativeEvaluationStrategyFactory.createEvaluationStrategy` returns `LmdbNativeEvaluationStrategy` (a `StrictEvaluationStrategy`) and picks `LmdbQueryOptimizerPipeline` when statistics support join estimation (LmdbNativeEvaluationStrategyFactory.java:31-51).
2. **Native source extraction**: the strategy walks the dataset stack via `SailDatasetTripleTermSource.getDatasets(NativeLmdbQuerySource.class)` (SailDatasetTripleTermSource.java:74-105). The walk looks through `UnionSailDataset` (explicit+inferred, UnionSailDataset.java:45-51) and transparent `DelegatingSailDataset` wrappers (close-suppression wrappers at SailSourceBranch.java:284,504,516); it stops (returns `List.of()`) at anything else. Results are merged by `CompositeNativeLmdbQuerySource.combine`, which requires identical `idSpace()` (the shared `ValueStore`; LmdbSailStore.java:1736) across sources (CompositeNativeLmdbQuerySource.java:30-45).
3. **Interception point**: `precompile(TupleExpr, QueryEvaluationContext)` is the per-node dispatch of `DefaultEvaluationStrategy` (DefaultEvaluationStrategy.java:459), so the override at LmdbNativeEvaluationStrategy.java:46-60 fires for every algebra node: first `LmdbNativeAggregateCompiler.tryCompile` (only `Group` or `Filter(Group)` roots, LmdbNativeAggregateCompiler.java:95-105), then `LmdbNativeQueryCompiler.tryCompile`, else `super.precompile`. Unsupported parents fall back generically while supported subtrees still compile natively.
4. **BGP compile**: `Builder.collect` accepts only `Filter`/`Join`/`StatementPattern`/`SingletonSet` (LmdbNativeQueryCompiler.java:112-136). Vars→slots (max 60, line 77,146); constants→IDs via `ValueStore.getId` at compile time (line 237); FROM/FROM NAMED become per-pattern `ContextConstraint` ID arrays (lines 184-223); unknown constant → `NativePlan.empty()` (correct empty result, not fallback). Conjunctive filters split, compiled to raw-ID filters (`SameTerm`, EQ/NE on IRI/BNode constants, `isIRI`/type tests decoding type bits from the ID itself via `ValueIds.getIdType`, lines 264-341,898-919) or `GenericNativeFilter` wrapping a precompiled `Predicate<BindingSet>` over the lazy `RowBindingSetView` (lines 246-262,792-810).
5. **Execution**: per `evaluate(bindings)` a `NativePlanIteration` seeds slots from base bindings (IDs passed through directly when the base is itself a `RowBindingSetView` of the same id-space, lines 1116-1121), greedily orders patterns by bound-term score then static estimate (lines 1142-1171), assigns each filter the earliest depth where its slot mask is covered (lines 1192-1218), then runs an index nested-loop with a mutable `long[] slots` + undo trail (`RowState.bind/mark/rollback`, lines 1295-1316). Fully-bound patterns degrade to `source.has()` existence probes (lines 617-648). Results are `RowBindingSetView` snapshots that resolve `Value`s lazily via `ValueStore.getLazyValue` (lines 1328-1332,1459-1465).
6. **Store bridge**: `LmdbSailDataset implements NativeLmdbQuerySource` (LmdbSailStore.java:1677); `statements()` takes a `StampedLongAdderLockManager` read lock held for the iterator's lifetime so `close()` (write lock) can't kill the LMDB txn mid-read (lines 1741-1757,1826-1866). `LmdbRecordIterator` now does exact-match `mdb_get` when all 4 key positions are bound (LmdbRecordIterator.java:107-141,255) and prefix-range cursor scans otherwise (replacing develop's `GroupMatcher` post-filtering).
7. **Generic-path EXISTS**: `ExistsQueryValueEvaluationStep` short-circuits statement-pattern subqueries to `StatementPatternQueryEvaluationStep.hasStatement` (ExistsQueryValueEvaluationStep.java:33-35), which uses the new `TripleSource.hasStatements` default (TripleSource.java:80-86) → `SailDataset.hasStatements` → `LmdbSailStore.hasStatementIterator` (LmdbSailStore.java:848) → `TripleStore.hasTriples` first-record probes (TripleStore.java:538-563).

## Perf-relevant design decisions

- Rows are primitive `long[]` slots; join backtracking uses a trail instead of copying (LmdbNativeQueryCompiler.java:1272-1316). One `long[]` copy + view object per emitted row (line 1328).
- Value materialization deferred to `getLazyValue`; `LmdbStatementIterator` adds a 4-entry MRU id→Value cache for the generic path (LmdbStatementIterator.java:39-116).
- Composite source pre-resolves active branches once; per-probe it lazily concatenates per-source iterators to avoid re-scanning (CompositeNativeLmdbQuerySource.java:63-67, comment cites "tens of thousands of small correlated LMDB probes").
- Fully-bound lookups use `mdb_get` (no cursor) and cursor/txn pooling elsewhere (LmdbRecordIterator.java:141-176); dedicated SPO-any-context and stat-based total-count probes (TripleStore.java:566-612).
- Write-path spin loops replaced `Thread.yield` with `onSpinWait`+periodic yield (`idleSpin`, LmdbSailStore.java:~160), power-of-two `CircularBuffer` with monotonic head/tail.

## Weaknesses/opportunities

- **Suspected correctness bug**: `SlotBoundFilter.requiredMask()==0` (LmdbNativeQueryCompiler.java:821-824) places `FILTER(bound(?x))` at depth 0 (line 1198-1200), evaluated once before any pattern binds `?x` (line 1065) → `SELECT * { ?x ?p ?o FILTER(bound(?x)) }` would return empty natively. Unverified (Routine C, no tests run).
- `RowBindingSetView.slot(name)` is a linear scan per `getValue`/`hasBinding` (LmdbNativeQueryCompiler.java:1446-1453); hot for `GenericNativeFilter` and downstream consumers of snapshots; no per-slot memoization of materialized `Value`s (line 1459-1465).
- Join order uses only compile-time static estimates with constants (line 178-181); no bound-slot-aware re-estimation, so score ties on star joins fall back to static cardinality; order/frames/filter assignment redone per `evaluate()` call — costly when the plan is the inner arg of a generic join (lines 1006-1021,1062-1067).
- Every native probe pays a read-lock acquire/release plus a wrapper allocation (LmdbSailStore.java:1741-1757); `estimate()` skips the lock entirely and races `close()` (line 1785-1795, masked by catch→+Inf).
- Failed compiles are retried per subtree; each attempt re-resolves constant IDs through `ValueStore.getId` (LmdbNativeQueryCompiler.java:237,421-427) — O(nodes × constants) LMDB lookups during precompile of unsupported queries.
- No `LeftJoin`/`Union`/`Optional`/property-path support in the BGP compiler (line 112-136) — the stated factorized-processing goal needs these; aggregate compiler only fires on `Group` roots (LmdbNativeAggregateCompiler.java:95-101).
- `nativeEnabled` system property parsed per query (LmdbNativeEvaluationStrategy.java:42).

## Fallbacks (generic RDF4J evaluation)

1. `-Drdf4j.lmdb.nativeQueryEngine.enabled=false` (LmdbNativeEvaluationStrategy.java:42,47).
2. `nativeSource == null`: triple source isn't `SailDatasetTripleTermSource`, or dataset walk hits a non-transparent wrapper — **uncommitted transaction changes** (`SailDatasetImpl` changeset stack, SailSourceBranch.java:527), serializable-isolation `ObservingSailDataset` (`isStatementAccessTransparent()==false`, ObservingSailDataset.java:46-49), or any foreign dataset (SailDatasetTripleTermSource.java:81-105). Native never sees uncommitted data — it bails instead.
3. Mixed id spaces across union branches (CompositeNativeLmdbQuerySource.java:36-39).
4. `!hasStatementsInSource()` — e.g. inferred-only view with `mayHaveInferred==false` (LmdbSailStore.java:1798-1800).
5. Unsupported algebra in fragment (`collect` false), filter-only/SingletonSet fragments (LmdbNativeQueryCompiler.java:142-145), or >60 slots (line 146).
6. Aggregate shapes other than `Group`/`Filter(Group)`, or unsupported aggregate/child algebra (LmdbNativeAggregateCompiler.java:95-105,135+).
7. Per-plan empty-result short-circuits (unknown constant IDs, empty FROM sets) return `NativePlan.empty()` — native empty result, not generic fallback (LmdbNativeQueryCompiler.java:138-141,166-167,219-221).

### optimizer
## Purpose

This branch's optimizer stack chooses join orders for LMDB (and generically) using DataSketches-based cardinality estimation, and feeds those orders to the native ID-slot executor. Components: `QueryJoinOptimizer` (generic heuristic orderer, used when sketches are unavailable), `SketchBasedJoinEstimator` (tuple-sketch statistics, 9.7k lines, mostly pre-existing on develop), `SketchJoinOrderPlanner` + `ParetoJoinMemoPlanner` (cost-based order search), `LmdbSketchJoinOptimizer` (LMDB-side segmenting/rewriting driver), `LmdbEvaluationStatistics` (glue, `JoinOrderPlanner`/`JoinFactorCostModel` SPI).

Changes vs develop are targeted:
- `QueryJoinOptimizer.java` (+381/-52): costing rework — `CARDINALITY_FLOOR=5` extracted (`:17`), fully-bound patterns no longer collapse to cost 1 (min exponent `0.25`, `:516`), foreign-var bonus capped at 8 (`:538`), `/nonConstantVarCount` softened to `/sqrt(n)` for n>1 (`:528`), repeated vars deduplicated (`countUniqueNonConstantVars`, `getUniqueVars`, `:579-616`), cartesian penalty `currentHighestCost` now reset per join scope and restored (`:30-34,80-81`) and updated from result-size too (`updateCurrentHighestCost`, `:564`). New behavior: candidates sharing runtime-bound vars are preferred in `selectNextTupleExpr` (`:386-389`) and in `selectBestStartingExpr` (`:150-158`); pairwise `reorderJoinArgs` now refuses disconnected (cross-product) candidates when a connected one exists (`:113-133,167-176`); multi-row VALUES are removed before ordering and re-inserted at first use (`placeBindingSetAssignments`, `:264-328`), singleton VALUES hoisted first.
- `SketchJoinOrderPlanner` (+60): singleton-VALUES factors get a dedicated mask; they are forced as the first legal seed/candidate (`firstLegalSingleBindingSetAssignment` `:966`, `preferSingleBindingSetAssignmentCandidates` `:2971`) and excluded from finite-seed delay logic.
- `LmdbSketchJoinOptimizer` (+72): `hoistSingleBindingSetAssignments` (`:2588`) applied to every ordered-segment path, including fallbacks.
- `FilterSelectivityKeys` (+74): memoizes origin-pattern sets, condition vars and local roles in query-model metadata (`:9-14,77-86,102-125`) — pure optimizer-time allocation savings.
- `SketchBasedJoinEstimator` (+27): pre-sizes the sketch cache directory before persisted-index load (`ensureCapacityForEntries`, `tableSizeForLoad`) and uses an allocation-free `findOrAdd` overload.
- `QueryEvaluationUtil` (+113): fast `long` parse/compare for integer literals (`parseLongFast`, `compareIntegerLiterals`) avoiding `BigInteger`, plus an 8-entry thread-local weak/soft calendar cache — hot path for filter evaluation, including the optimizer's filter-selectivity sampling and the native executor's complex filters.

## Mechanics

1. Pipeline: `LmdbNativeEvaluationStrategyFactory:47` picks `LmdbQueryOptimizerPipeline` only when `statistics.supportsJoinEstimation()` (= `SketchBasedJoinEstimator.isReadyNonBlocking()`, `LmdbEvaluationStatistics:94`); otherwise the standard pipeline with `QueryJoinOptimizer`.
2. `LmdbSketchJoinOptimizer.optimize` visits joins, splits into segments, and calls `statistics` as a `JoinOrderPlanner` (`:2535`). `LmdbEvaluationStatistics.planJoinOrderAttempt` may try bounded-GREEDY before DP for large action counts (`shouldTryBoundedGreedyFirst`).
3. `SketchJoinOrderPlanner` builds bitmask `PlanFactor`s (var masks as `long`s, ≤64 vars/factors) and dispatches (`:328`): `DYNAMIC_PROGRAMMING` → `ParetoJoinMemoPlanner.planMemo()` — subset-DP with a Pareto frontier (limit 8) per memo group; `GREEDY` → `planBeam()` beam search, width 8. Cost is the 5-dim `JoinCostVector` (totalWork, finalRows, maxIntermediate, uncertainty, cartesianWork; `JoinCostVector.java:23-27`), pruned by Pareto dominance. DP is refused above 20 factors (`MAX_DYNAMIC_PROGRAMMING_JOIN_ARGS`, `:318`).
4. Sketches: Apache DataSketches `ArrayOfDoublesSketch` (theta tuple sketches with a signed multiplicity summary) over S/P/O/C singles, complement-degree sketches, and all six component pairs; constants are hashed into per-component buckets (`hash()`, `SketchBasedJoinEstimator:3920` — `hashCode % bucketCount`). Pattern cardinality = theta-scaled positive multiplicity sum (`TupleSketchOps.estimatePositiveSum`); deletes as negative summaries (add/delete sketch pairs netted). Join size = sketch intersection over the shared variable, summing products of multiplicities (`intersectProductStats`, used in `estimateJoinStep`/`JoinEstimate.join`, `:3080-3100, 4642-4720`); no-sketch fallback is `leftRows*rightRows/max(distinct)` (`estimateStatsSharedVarJoinRows`, `:4918`), plus an exact-lookup fallback when intersections estimate 0 (`zeroIntersectionFallbackJoinRows`, `:4665`).
5. Per-step physical cost comes from `LmdbEvaluationStatistics.estimateFactorCost` (`:347-470`): sketch `AccessShape.estimateAccessRows(prefixMask)` models LMDB index access rows for the bound-component mask, direct-lookup detection, nested-invocation multiplication (`accountNestedInvocationWork`) capped by predicate-scan bounds.
6. Filter selectivity: `estimateFilterPass` (`SketchBasedJoinEstimator:4135`) cascades learned runtime stats (`JoinStatsProvider` keyed by `FilterSelectivityKeys` pattern/filter/template keys) → sampled (`LmdbFilterSelectivityStats`, which evaluates conditions on sampled rows — hence the `QueryEvaluationUtil` fast paths) → structural heuristics. Pass ratios multiply step outputs and drive filter-placement "actions" inside the planner's state space (`plannedFilterActionCount`, deferred-filter masks).
7. The native executor **re-derives order at runtime**: `LmdbNativeQueryCompiler` flattens Join/Filter trees (`collect`, `:113`), then `chooseOrder()` (`:1142-1171`) greedily picks by `score = Σ boundScore` (constant=2, bound slot=1, else 0; `:506,587`) with the compile-time `source.estimate(...)` (constants only, `:179`) as tiebreak; `assignFilters()` places filters at earliest bound depth.

## Perf-relevant design decisions

- Bitmask planning state (`long` var/factor masks, array memos ≤16 factors / 2^20 groups, `SketchJoinOrderPlanner:62-64`); `SmallVarStatsMap` with shared variable dictionary to avoid HashMap churn; per-query `OptimizationScope` caching sketch intersections.
- Metadata-attached memoization in `FilterSelectivityKeys` avoids repeated `StatementPatternCollector`/`VarNameCollector` walks across optimizer passes.
- Sketch directory rehash pre-sizing avoids repeated rehash during persistence load.
- `QueryEvaluationUtil` avoids `BigInteger`/`XMLGregorianCalendar` allocation in comparisons.
- `QueryJoinOptimizer.reorderJoinArgs` memoizes pairwise `getCardinality(new Join(a,b))` (`:412-427`) but is still O(n²) sketch intersections per scope.

## Weaknesses / opportunities

- **Planner order discarded by native executor**: `chooseOrder` (`LmdbNativeQueryCompiler:1142`) ignores the sketch-planned order entirely; boundness (max 2 per term) dominates cardinality, and `staticEstimate` is computed with slot terms as UNKNOWN — a bound-but-huge-fanout pattern beats an unbound-but-tiny one; disconnected constant-rich patterns can seed cartesian expansion the planner deliberately avoided.
- Bucketed hashing (`hash()`, `SketchBasedJoinEstimator:3920`) conflates constants sharing a bucket — rare constants inherit hot-value cardinalities; `String.hashCode` is weak vs Murmur used elsewhere.
- Estimates lag data: incremental signed-multiplicity updates + background rebuild mean post-churn estimates drift; `isReadyNonBlocking` flips the whole pipeline per strategy creation.
- `QueryJoinOptimizer` min-exponent 0.25 (`:516`) makes fully-bound existence checks cost `card^0.25`, potentially delaying cheap O(1) LMDB point lookups that prune early.
- Runtime-bound candidate restriction (`:386-389`, `:150-158`) can force a high-fanout bound pattern before a highly selective constant pattern.
- Singleton-VALUES hoisting in `LmdbSketchJoinOptimizer:2546-2549` discards `plannerFilterPlacementSteps` whenever hoisting changes order, losing planned filter depths.
- `hasSharedVariable` recomputes `getBindingNames()` sets repeatedly inside O(n²) loops (`QueryJoinOptimizer:184-233`).

## Fallbacks

- Sketches not ready → `StandardQueryOptimizerPipeline` + heuristic `QueryJoinOptimizer` (`LmdbNativeEvaluationStrategyFactory:47`, `LmdbEvaluationStatistics:94`).
- Planner rejection paths (`SketchPlannerPath`, `SketchBasedJoinEstimator:371-378`): NOT_READY, UNSUPPORTED_CYCLE/WRAPPER/MULTI_SHARED_VAR/SHAPE, factor rejection (non-pattern factors), >20 factors for DP (`SketchJoinOrderPlanner:318`) → `LmdbSketchJoinOptimizer.locallySelectiveFallbackOrder` (`:2729`) keeps input order (plus singleton-VALUES hoist).
- Segments <2 args or statistics not a `JoinOrderPlanner` → passthrough (`LmdbSketchJoinOptimizer:2534`).
- Native compile bails to generic RDF4J evaluation (`tryCompile` returns null) on: any non Join/Filter/StatementPattern/SingletonSet node (`LmdbNativeQueryCompiler:113-136`), >60 slots, uncompilable filters, pattern-only-free fragments (`:140-156`).
