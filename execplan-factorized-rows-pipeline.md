# Full-pipeline factorized execution for native LMDB row queries (migration steps 5–7)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `.agent/PLANS.md` (from the repository root).

## Purpose / Big Picture

The LMDB store has a native query engine that evaluates SPARQL fragments directly over 64-bit value ids ("slots") instead of materialized RDF values. Aggregate queries already benefit from *factorized* execution at the tail of the join order (`LmdbNativeFactorizedTail`, measured 11.6x–48x on star shapes at fan-out 100), but ordinary SELECT queries still enumerate every combination of pattern matches one row at a time: for `SELECT ?s WHERE { ?s :p1 ?a . ?s :p2 ?b }` the engine produces `kₐ×k_b` physical rows per `?s`, probing the `:p2` branch once per *enumerated* `(?s,?a)` row, even though nothing downstream distinguishes those rows. After this change, the trailing run of join positions whose fresh variables no later operator consumes is evaluated *once per distinct probe key* as counted/valued batches: variables the projection keeps are enumerated from an in-memory batch (an odometer over per-branch value lists instead of nested storage probes), projected-away variables degrade to a per-key match count (multiplicity) under plain SELECT, and to a first-match existence probe (semi-join) under SELECT DISTINCT — the "existential skip" of the design document. The flat prefix of the join order additionally pulls its quads through batched `fill` calls (one lock acquisition and JNI-amortized cursor walk per up-to-1024 quads) instead of per-row `next()`.

Observable outcome: SELECT queries over star and path shapes return identical multisets of solutions to the generic RDF4J evaluator (verified by differential tests that run each query twice, once with `rdf4j.lmdb.nativeQueryEngine.enabled=false`) while executing measurably faster on a new JMH star benchmark for row (non-aggregate) queries; a package-visible engagement counter proves the fast path actually runs.

## Surprises & Discoveries

- Observation: the "aggregate" compiler is the full native SELECT engine — `LmdbNativeAggregatePlanner.compileRowRoot` compiles `[Slice] [Distinct|Reduced] Projection [Order]` roots into `NativeRowsStep`, so projection/DISTINCT metadata needed for existential skip is already at hand; the older `LmdbNativeQueryCompiler` only handles bare BGP fragments embedded in generic plans.
  Evidence: LmdbNativeAggregatePlanner.java:147-240, LmdbNativeRowStep.java:143-357.
- Observation: the helpers this plan needs already exist from the factorized-tail work: `MultiJoinPlan.derivedPlan(RowState)` and `openChain(OrderedPlan, upToExclusive, RowState)` (LmdbNativeJoinPlans.java:144-175), `PatternPlan.freshProducedMask/quadPositionOfSlot/hasRepeatedSlot/rejectsNullContextAtBind` (LmdbNativePatternPlan.java:219-251), `PatternCursor.fill` (LmdbNativePatternTerms.java:258-286), retained probes via `NativeLmdbQuerySource.newProbe()`.

## Decision Log

- Decision: integrate at `NativeRowsStep.evaluateAll` (the row-root engine) rather than rewriting `LmdbNativeQueryCompiler`'s DFS.
  Rationale: only the row root sees projection/DISTINCT (existential skip needs them); `NativeRowsStep` materializes results into a list anyway, so odometer emission is a local change; the bare-BGP engine's consumers re-enumerate at the iterator boundary, shrinking the win. The bare-BGP engine is recorded as a follow-up.
  Date/Author: 2026-07-06, Claude Code.
- Decision: keep the existing `openChain` cursor machinery (JoinCursor incl. uncorrelated replay) for the *flat prefix* in M2 and only replace the trailing unconsumed run; convert the prefix to chunked fills in M3 as a separately benchmarkable step.
  Rationale: JoinCursor's replay/materialize and probe reuse are proven by the existing suite; the tail is where enumeration is pure waste. Two smaller steps beat one risky rewrite.
  Date/Author: 2026-07-06, Claude Code.
- Decision: tail roles are ENUM (fresh slots intersect the projection), COUNT (projected away, non-DISTINCT: only the match count matters, the projected row is emitted `count`-product times), EXISTS (projected away under DISTINCT: first match suffices). Any ORDER BY forces fallback in v1 (order keys may be unprojected; treating them as outputs is a recorded follow-up).
  Rationale: COUNT preserves SPARQL solution multiplicity exactly; EXISTS is correct only when duplicates collapse, which DISTINCT guarantees (REDUCED compiles to a plain stream, so it takes the COUNT role path).
  Date/Author: 2026-07-06, Claude Code.
- Decision: per-tail-branch memoization keyed on the branch's read-slot values (mirroring `LmdbNativeFactorizedTailBranch`), with an entry cap after which probes bypass the memo.
  Rationale: prefix rows repeat probe keys (a hub appears once per prefix combination); memoization converts per-row probes into per-distinct-key probes, which is where the multiplicative ~f-per-level compression comes from.
  Date/Author: 2026-07-06, Claude Code.
- Decision: kill switch `rdf4j.lmdb.factorizedRows.enabled` (default on, read once at class load) plus a package-visible `ENGAGED` counter, mirroring the factorized-tail conventions.
  Rationale: A/B benchmarking across JMH forks, operator escape hatch, and test observability that the fast path engages rather than silently falling back.
  Date/Author: 2026-07-06, Claude Code.

## Outcomes & Retrospective

All milestones complete. Migration steps 5 and 6 of the factorization design now cover ordinary SELECT queries: the trailing unconsumed run of the join order evaluates factorized (34x on the DISTINCT semi-join star, 4x on multiplicity-only projection), the flat prefix pulls quads through batched fills with an adaptive per-key quad memo (chunked prefix, step 5's ScanOp/ExtendOp shape), and existential skip (step 6) falls out of the role assignment. The deliberate deviation from the design document — per-branch memoized batches plus an odometer instead of a materialized IdChunk/ChunkSet dataflow — kept the change additive and gated while delivering the same asymptotics on B-tree storage; the design's step 7 (ascending-key forward-walk/SIP inside `LmdbRecordIterator.reset`, then bitmap semi-joins) remains open, as do: ENUM value-batch spilling for pathological fan-outs (currently memo-bypassed and counted), ORDER BY as an output role, extending the factorized sink to `NativeGroupStep` prefixes, and the bare-BGP engine (`LmdbNativeQueryCompiler`) whose consumers re-enumerate anyway. Lesson repeated from the tail work: differential tests plus an engagement counter made a deep engine change safe to land in one day; the only surprises were harness-level (binding-order canonicalization, REDUCED's legitimate engine variance).

## Context and Orientation

All paths are repository-relative; the module is `core/sail/lmdb`, package `org.eclipse.rdf4j.sail.lmdb`. Terms: a *slot* is an index into `RowState.slots`, a `long[]` of LMDB value ids per query variable (`UNKNOWN` sentinel = unbound; `NULL_CONTEXT_ID` = the default graph). A *SlotPlan* (LmdbNativeSlotPlan.java:95) is the native plan node interface; `PatternPlan` (LmdbNativePatternPlan.java) is its triple-pattern leaf with four `Term`s (slot, constant, both, or unbound) and a `ContextConstraint`; `MultiJoinPlan` (LmdbNativeJoinPlans.java:95) is a reorderable bag of inner-join children plus `MaskedFilter`s (a `NativeBooleanFilter` with the bitmask of slots it reads); at open time it derives a join order and per-depth filter placement (`derivedPlan`, memoized per bound-slot mask) and builds a chain of `JoinCursor`s, each re-opening its right child per left row through a retained `NativeProbe`. `NativeRowsStep` (LmdbNativeRowStep.java:143) is the compiled `[Slice][Distinct|Reduced]Projection` root: `evaluateAll` seeds a `RowState` from the caller's bindings, opens `arg.open(row)`, and per cursor row optionally dedups on the projected id tuple (`registerDistinct`) and materializes a `QueryBindingSet` (`project`). `PatternCursor.fill(long[] buffer, int maxRows)` (LmdbNativePatternTerms.java:258) drains up to maxRows quads (4 longs each) per call, amortizing lock/JNI costs; a return of 0 means exhausted (callers loop, batches may end early at fixed-context boundaries). The differential-test oracle is the system property `rdf4j.lmdb.nativeQueryEngine.enabled=false`, which routes the same query through the generic evaluator.

The waste this plan removes: in `evaluateAll`'s loop every solution of the trailing patterns costs a `JoinCursor.next()` → probe open → `bind`/rollback → projection, even when the row differs from its predecessor only in variables nobody downstream distinguishes (projected away, or enumerable from one in-memory batch).

## Plan of Work

Milestone 1 — shape planning and gates. New file `core/sail/lmdb/src/main/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeFactorizedRows.java` with a static factory `tryCreate(MultiJoinPlan plan, MultiJoinPlan.OrderedPlan derived, RowState row, long seedMask, int[] sourceSlots, boolean distinct)` that computes, over the derived order: `boundBefore[d]` (seed plus cumulative produced masks), `fresh[d] = order[d].freshProducedMask(boundBefore[d])`, pattern read masks (`producedMask & boundBefore[d]`), and filter masks at their assigned depths. Walking backwards it finds the *flat prefix length m*: the smallest m such that for every tail depth d ≥ m, `fresh[d]` is disjoint from every later pattern's read mask and from every filter mask at depths ≠ d (each tail filter may read only its own chunk's fresh slots plus prefix slots). Gates (any failure returns null → the existing enumerating loop runs): every ordered child is a `PatternPlan`; no tail pattern has a repeated fresh slot (`hasRepeatedSlot` over fresh terms); no ORDER BY (checked by the caller); tail depth count ≥ 1 and the tail is estimated useful (v1: always when m < n). Roles per tail depth: ENUM if `fresh[d] ∩ outputMask ≠ 0` (outputMask = bits of `sourceSlots`), else EXISTS if `distinct`, else COUNT. The factory records, per tail branch: the `PatternPlan`, its read-slot list (probe key), the fresh slots and their quad positions (`quadPositionOfSlot`), the role, and `rejectsNullContextAtBind`.

Milestone 2 — tail execution. Same file: `evaluate(RowState row, TailSink sink)` — for each prefix row produced by `plan.openChain(derived, m, row)` (filters at prefix depths apply inside the chain as today), evaluate each tail branch once: build the probe key from the branch's read slots, look it up in the branch memo (`HashMap<GroupKey, TailResult>`; `GroupKey` from LmdbNativeAggregateState.java); on miss, open the pattern through a per-branch retained `NativeProbe` via `PatternPlan.openRaw(row, probe)` and consume with `PatternCursor.fill` into a `TailResult`: EXISTS pulls batches of size 1 and stops at the first accepted quad (count 0/1); COUNT counts accepted quads across full batches; ENUM appends the accepted quads' fresh-slot values (stride = fresh slot count) to a growing `long[]`. "Accepted" = passes the null-context rejection when `rejectsNullContextAtBind`, and passes every filter assigned to that tail depth, evaluated per quad by writing the quad's fresh values into `row.slots` directly, calling `filter.accept(row)`, and restoring `UNKNOWN` (the planning gate guarantees those filters read nothing else non-prefix). If any branch's count is 0 the prefix row contributes nothing (inner-join semantics). Emission into `NativeRowsStep.evaluateAll`'s results list: multiplicity `mult` = product of COUNT-branch counts; the odometer enumerates the cross product of ENUM branches, writing each combination's values into `row.slots` (direct writes, restored after), then `project(row.slots, values)` once per combination, added `mult` times under plain SELECT (the same immutable `QueryBindingSet` instance is reused); under DISTINCT each combination goes through `registerDistinct` first and EXISTS branches contribute nothing (already 0/1). The memo caps at 65,536 entries (beyond: probe without memoizing); ENUM results larger than 1<<20 values bypass the memo too (recorded, not silently dropped: a counter tracks bypasses). Honor `streamingCap` by checking the emit cap between odometer steps. Wire-up in `LmdbNativeRowStep.evaluateAll`: in the `orderSlots.length == 0` branch, after `arg instanceof MultiJoinPlan` and the kill switch check, try `tryCreate`; on success run the factorized loop instead of the plain cursor loop.

Milestone 3 — chunked flat prefix. Replace, for the factorized path only, the prefix `openChain` with a chunk-per-depth loop in the same file: each prefix depth holds a `long[4*1024]` quad batch filled via `PatternCursor.fill`, a position, and the depth's filters; advancing a depth means `bind`ing the quad at the current position (mark/rollback per quad as `PatternRowCursor` does) and refilling when the batch is exhausted. Uncorrelated depths (read mask disjoint from all earlier fresh masks and the seed-varying part) memoize their accepted quads once (cap 65,536 rows, as `JoinCursor.MAX_MATERIALIZED_ROWS`) so the replay optimization is not lost. This milestone is behavior-neutral relative to M2 and separately benchmarkable.

Milestone 4 — measurement. New JMH benchmark `core/sail/lmdb/src/test/java/org/eclipse/rdf4j/sail/lmdb/benchmark/FactorizedRowsStarBenchmark.java` modeled on `FactorizedTailStarBenchmark` (same store shape: hubs × fan-outs), with SELECT variants: `selectHubDistinct` (both tails EXISTS — semi-join), `selectHub` (COUNT multiplicity), `selectTails` (ENUM odometer), `selectTwoHopLeaf` (mid-plan chunk, exercises M3), each verifying result counts per invocation; toggled via `-Drdf4j.lmdb.factorizedRows.enabled`. Spot-check `ThemeQueryBenchmark` MEDICAL_RECORDS before/after.

## Validation and Acceptance

New test class `core/sail/lmdb/src/test/java/org/eclipse/rdf4j/sail/lmdb/LmdbNativeFactorizedRowsTest.java` builds a star dataset (hubs with two fan-out predicates of differing cardinalities, a hub with zero matches on one branch, IRIs and literals among fan-out values, one triple in a named graph) plus a two-hop chain, and asserts for each query that the native result multiset equals the generic result multiset (`rdf4j.lmdb.nativeQueryEngine.enabled=false`, restored in finally; compare `List<BindingSet>` as multisets of projected maps). Queries cover: SELECT of the hub only (COUNT role — multiplicities must multiply); SELECT DISTINCT of the hub (EXISTS role — zero-match hubs vanish, no duplicates); SELECT of tail variables (ENUM odometer); SELECT mixing prefix and tail variables; a FILTER on one tail variable (per-quad tail filter); a FILTER spanning two tail variables (gate → those chunks flatten or fall back); LIMIT without ORDER BY (emit cap mid-odometer); a named-graph tail pattern; a repeated-variable tail pattern (gate → fallback); correlated evaluation (the step under a generic join seeding bindings); REDUCED (treated as plain stream). Engagement: `LmdbNativeFactorizedRows.ENGAGED` (package-visible `AtomicLong`) must grow across the native runs of non-gated queries and stay flat for gated ones. Acceptance: the new class passes, `python3 .codex/skills/mvnf/scripts/mvnf.py core/sail/lmdb` unit summary stays green, and the M4 benchmark shows the ON column beating OFF on the star variants at fan-out 100.
